# 产品介绍

## 产品定位

Pet Mobility Agent 是一个面向城市养宠人的宠物友好地图 Agent。它不是通用地图的宠物皮肤，也不是单纯的遛狗路线规划器，而是基于宠物档案、常走路线、天气环境、社区热力、UGC POI 和宠物分身，帮助主人判断今天该不该出门、走哪条熟悉路线、是否适合探索新地点、路上可能遇到什么，以及哪些宠物友好地点在这一次真实可用。

一句话定位：

> 普通地图给路线，Pet Mobility Agent 给适合这只宠物的附近判断。

本产品的核心不是“附近有什么宠物友好地点”，也不是“生成一条最短路线”，而是回答：

- 今天这只宠物适合出门吗？
- 如果只是日常遛狗，常走路线里哪条今天更合适？
- 如果有明确目标，哪条路线和哪个 POI 更适合这只宠物？
- 路上会不会更容易遇到其他狗、人流、电动车、高温、施工或禁入点？
- 地图上的宠物友好信息是否可信、是否过期、是否适合本次出行？
- 用户和社区的反馈如何沉淀为下一次推荐的依据？

因此，Pet Mobility Agent 更像一个“宠物附近决策层”：

```text
宠物友好地图 Agent
├─ 宠物档案与宠物分身
├─ 常走路线记忆
├─ 有目标推荐
├─ 无目标出门建议
├─ UGC 宠物友好 POI
├─ 社区路线热力
├─ Agent 推荐解释
└─ 隐私与可信度机制
```

本 PRD 第 3 版将产品主线从“带宠出行任务规划器”升级为“宠物友好地图 Agent”。路线规划仍然重要，但它被降级为功能输出；真正的产品核心是把宠物画像、个人路线习惯、附近 POI、社区趋势和 Agent 判断组织成一个可持续学习的出门决策系统。

## 产品命名口径

PRD 内部继续使用 `Pet Mobility Agent` 作为项目名。对外可考虑以下表达：

# 背景

## 市场与用户背景

中国宠物市场正在从“养宠消费”进入“养宠生活方式基础设施”阶段。宠物食品、医疗、洗护、美容、用品、保险和智能硬件等供给逐渐成熟，城市养宠人愿意为宠物健康、体验、便利性和情绪价值支付更多费用。

对于城市养狗人来说，宠物已经不只是家庭里的消费对象，而是日常时间安排、居住选择、周末活动、本地服务和城市空间使用方式的一部分。用户不是偶尔需要一个“宠物友好地点”，而是每天都要做一系列小判断：

- 今天热不热，能不能走远？
- 常走的那条线现在会不会狗很多？
- 这家店到底能不能带狗进去，还是只能坐户外？
- 路上有没有水碗、草地、阴影、施工、禁入或风险点？
- 如果今天狗状态紧张，是不是应该换一条更安静的路线？
- 如果想顺路买粮、去医院或找咖啡店，哪条路线最稳？

当前市场上已经出现了几类相关供给：

| 供给类型 | 示例 | 当前不足 |
| --- | --- | --- |
| 通用地图 | 高德、百度、Google、Apple Maps | 理解道路和交通，但不理解宠物需求。 |
| 本地生活平台 | 大众点评、小红书、团购平台 | 有宠物友好信息，但多为非结构化评论，可信度和时效性弱。 |
| 宠物服务平台 | 洗护、寄养、训练、上门喂养 | 解决单点服务，不解决日常出门判断。 |
| 宠物社区 | 晒宠、经验分享、地点推荐 | 内容丰富但难以进入地图和推荐系统。 |
| 运动记录工具 | Strava、Keep 等 | 记录人的运动，不理解宠物约束和附近宠物友好信息。 |
| 智能硬件 | GPS、项圈、摄像头、喂食器 | 记录位置或健康数据，但不判断出门路线是否合适。 |

Pet Mobility Agent 的机会是把这些分散能力连接起来，让用户每天打开产品时不只是“找地方”，而是获得一句有依据的附近判断：

> 现在这只宠物，更适合走哪条熟悉路线，或者去哪个宠物友好地点。

## 核心问题

城市养宠人的真实问题不是“不知道怎么走”，而是：

- 熟悉几条路线，但不知道今天哪条更合适。
- 想避开高温、暴雨、施工、车流、电动车、流浪狗、狗多区域、禁入区域。
- 想知道附近哪些地方真的宠物友好，而不是点评里模糊写一句“可带宠”。
- 希望系统记住自家宠物的习惯，例如怕大狗、不能走太久、喜欢草地、不爱人多、需要顺路买粮。
- 愿意贡献 POI 或路线反馈，但前提是隐私安全、有反馈、有价值。
- 想要一个有情感连接的宠物入口，而不是冷冰冰的数据面板。

现有方案的断点在于：

- 普通地图理解人和车的路线，但不理解宠物的体力、情绪、社交偏好和天气风险。
- 宠物友好地图如果只是 POI 列表，会很快变薄，因为“宠物友好”不是一个稳定的 yes/no 标签。
- 用户常走路线非常重要，但大多数地图不会把“熟悉路线今天是否适合这只宠物”作为核心问题。
- 社区里有大量附近经验，但没有结构化沉淀为可计算的路线热力、POI 可信度和风险图层。
- 虚拟宠物如果只是装饰，会成为一次性玩具；如果承载宠物画像和路线反馈，则可以成为产品的情感入口和解释界面。

## 类比产品启发

本产品可以借鉴几个成熟产品的机制，但不能照搬。

| 类比产品 | 可借鉴点 | 不应照搬 |
| --- | --- | --- |
| Strava | 路线记录、热力图、个人历史、社区数据聚合。 | 竞技化排行榜、速度里程优先、暴露敏感路线。 |
| Waze | 用户上报、风险点、社区共建地图、动态提醒。 | 强实时导航和看见他人位置，宠物场景早期应做趋势和概率。 |
| AllTrails | 路线详情、难度、环境标签、用户评价、适合人群。 | 户外徒步的长路线范式，日常遛狗更多是 routine 和附近微判断。 |
| 大众点评 / Yelp | POI 评价、图文、标签、最近验证、商户信息。 | 非结构化评论搜索，宠物友好规则必须结构化。 |
| 宠物社区 | 宠物身份、经验分享、附近同好。 | 泛社区信息流，早期社区应围绕地图贡献和路线反馈。 |

## 机会判断

Pet Mobility Agent 的真正机会不在“路线规划”，而在“附近可信判断”：

```text
用户熟悉路线
  + 宠物有个体差异
  + 附近信息会变化
  + 社区反馈可以聚合
  + Agent 能做解释和推荐
  + 宠物分身提供情感入口
= 懂这只宠物的附近地图
```

# 产品原则

## 原则一：先判断适不适合，再生成路线

系统不能默认给出路线。面对高温、狗群密度高、禁入、交通刺激、宠物状态不适合等情况时，Agent 应先做风险门控，再推荐路线或替代方案。

## 原则二：优先理解常走路线，而不是强推探索

大多数用户日常遛狗会走熟悉路线。产品应先记住并理解常走路线，再在天气、时间、社区热力和宠物状态变化时给出调整建议。

## 原则三：宠物友好不是标签，而是结构化规则

一个 POI 不能只写“宠物友好”。至少应描述可进入范围、体型限制、牵引要求、是否有水、是否有户外位、最近验证时间、争议和置信度。

## 原则四：社区数据必须先保护隐私，再产生价值

热力图只能表达趋势和概率，不能展示实时个人位置、单条原始路线或可推断家庭地址的起终点。

## 原则五：宠物分身必须服务决策

Q 版/3D 宠物形象不是装饰。它应承载宠物画像、路线偏好、天气风险、遇狗概率和反馈引导，让用户更直觉地理解推荐。

## 原则六：Agent 输出必须可解释、可执行、可反馈

每次推荐都要说明为什么推荐、为什么不推荐、风险是什么、备选是什么、用户执行后如何反馈。

# 目标用户与使用场景

## 主要用户

主要用户是居住在中国大陆城市中的养狗人，尤其是以下几类：

- 城市日常遛狗人：每天走熟悉路线，希望知道今天哪条更合适。
- 敏感犬或反应犬主人：需要避开狗、人流、电动车、儿童、噪声和复杂路口。
- 老年犬、幼犬、短鼻犬或慢病犬主人：需要更保守的天气、距离和强度控制。
- 新手养狗人：不知道什么样的路线、地点和时段更安全。
- 新搬家宠物家庭：对附近宠物友好地点、风险点和狗友路线缺少认知。
- 喜欢探索宠物友好地点的用户：想去咖啡店、公园、草地、宠物店、医院或服务点，但需要更可信的宠物规则。
- 愿意贡献地图数据的核心狗友：愿意上传 POI、纠错、标记狗多和风险点。

## 次要用户

次要用户包括宠物服务和本地空间供给方：

- 宠物友好餐饮、商圈、社区商业。
- 宠物医院、宠物店、洗护店、训练师、寄养机构。
- 社区物业、园区、商场和公园管理方。
- 宠物智能硬件和保险服务方。

## 核心使用场景

| 场景 | 用户问题 | 产品回答 |
| --- | --- | --- |
| 无目标日常出门 | 准备带狗下楼，今天走哪条常走路线？ | 结合宠物档案、天气、社区热力和历史反馈，推荐熟悉路线或降级路线。 |
| 有目标附近出行 | 想去咖啡店、宠物店、医院、公园或草地。 | 召回 POI 和路线，判断宠物适配度、路线风险和 POI 可信度。 |
| 高风险天气出门 | 高温、降雨、湿度高、地面烫、空气质量差。 | 判断是否适合出门，建议缩短、改时段、换路线或去室内。 |
| 避狗或社交 | 用户想避开狗，或想让宠物适度社交。 | 根据社区热力和宠物画像推荐安静路线或社交路线。 |
| 地图贡献 | 用户发现水碗、草地、禁入、狗多或宠物友好店。 | 结构化上传 POI 或风险点，进入待验证和推荐系统。 |
| 宠物分身生成 | 用户上传宠物照片，希望生成 Q 版/3D 形象。 | 生成宠物分身，并让它参与路线解释、反馈和社区身份。 |

# 功能优先级与范围

## 判断口径

本 PRD 不以“功能是否酷”判断优先级，而以它是否能支撑以下闭环判断：

```text
宠物画像 -> 常走路线/目标输入 -> 天气和地图上下文 -> 社区趋势与 POI 可信度 -> Agent 推荐解释 -> 用户执行 -> 反馈沉淀
```

优先级定义：

| 优先级 | 定义 | 判断问题 |
| --- | --- | --- |
| P0 必须闭环 | 没有它，产品无法证明“懂我家宠物的附近地图”成立。 | 是否直接支撑出门前推荐、路线记忆、POI 反馈和隐私控制？ |
| P1 产品化 | 让真实用户可以反复使用并形成社区数据飞轮。 | 是否提升热力图、可信度、互动、主动提醒和留存？ |
| P2 平台化 | 面向多城市、多服务方、商业化和复杂协作。 | 是否依赖规模数据、商家供给、硬件或复杂治理？ |
| 暂不进入范围 | 会稀释 MVP 或引入高风险履约/合规复杂度。 | 是否会让产品偏离宠物附近决策主线？ |

## P0：必须跑通的产品闭环

P0 目标是证明：用户愿意在出门前打开产品，让它基于宠物档案、常走路线、天气和 POI 判断“现在走哪条更合适”。

| 功能组 | P0 范围 | 验收信号 |
| --- | --- | --- |
| 宠物档案 | 体型、年龄、精力、怕热、怕狗、怕车、可走时长、社交偏好。 | 推荐结果能体现宠物差异。 |
| 宠物分身基础版 | Q 版 2D 头像；展示版可异步生成静态或轻 idle 3D。 | 宠物分身能表达天气、路线强度、遇狗概率等状态。 |
| 常走路线 | 用户手动保存或系统自动识别 2-5 条 routine。 | 用户打开产品时能看到“今天常走路线是否适合”。 |
| 出门 Agent | 支持有目标推荐和无目标常走路线判断。 | Agent 能输出推荐理由、风险和备选。 |
| 宠物友好地图 | 基础 POI、风险点、水源、草地、公园、医院、宠物店。 | POI 能进入推荐，而不是只展示在地图上。 |
| UGC POI | 上传、纠错、待验证、最近验证时间。 | 新 POI 进入待验证状态，并可被低置信引用。 |
| 路线反馈 | 走完后 1-2 个轻量问题，例如狗多吗、是否顺利、是否有风险。 | 反馈能影响下一次推荐或 POI 可信度。 |
| 天气与风险 | 温度、降雨、时间段、连续暴露、狗密度等基础风险。 | 高风险时能拒绝或降级路线。 |
| 隐私控制 | 路线记录开关、社区贡献 opt-in、起终点隐藏。 | 用户能明确控制是否贡献社区热力。 |

## P1：产品化与社区数据飞轮

| 功能组 | P1 范围 | 目标 |
| --- | --- | --- |
| 社区热力图 | 基于 opt-in 的脱敏聚合热力。 | 展示狗友常走区域、狗多时段、安静路线、风险趋势。 |
| 个人路线交叉分析 | 常走路线与社区狗密度、安静程度、风险点交叉。 | 提醒“这条路线 18:00-19:00 狗多”。 |
| POI 可信度体系 | 多人确认、过期降权、争议标记、商家规则变化。 | 提升宠物友好信息可信度。 |
| 宠物分身互动版 | 低面数 3D、idle/walk/sit/sniff、地图状态反馈。 | 成为可互动 route companion。 |
| 主动提醒 | 高温、降雨、空气质量、狗多时段、常走路线异常。 | 让用户形成出门前打开习惯。 |
| 弱社交 | 同路线狗友、附近搭子、但不暴露实时位置。 | 提升附近感和贡献动力。 |
| 路线标签 | 安静路线、嗅闻路线、短程路线、避狗路线、社交路线。 | 让路线更符合宠物需求语言。 |

## P2：平台化与商业化

| 功能组 | P2 范围 | 目标 |
| --- | --- | --- |
| 商家端 | POI 认领、宠物友好认证、活动发布。 | 提升 POI 真实性和商业合作。 |
| 服务交易 | 洗护、寄养、训练、宠物友好车。 | 从推荐进入服务履约。 |
| 可穿戴/硬件 | 智能项圈、GPS、健康数据。 | 将活动和健康信号纳入推荐。 |
| 高级宠物分身 | 多宠物、AR check-in、路线回放、装备/徽章。 | 提升留存和身份表达。 |
| 社区治理 | 地图贡献者等级、审核队列、区域志愿者。 | 支撑多区域数据质量。 |
| 多 Agent | 地图 Agent、隐私 Agent、推荐 Agent、商家核验 Agent。 | 在规模复杂后拆分职责。 |

## 暂不进入 MVP

以下内容不进入 P0：

- 泛社区信息流。
- 实时看到其他狗或狗主的位置。
- 排行榜和竞技化路线挑战。
- 完整商家后台。
- 真实交易、支付和服务履约。
- 完整 AR 宠物互动。
- 医疗诊断和行为训练诊断。
- 全国地图数据铺开。
- 与硬件深度绑定。

# 核心产品模块

## 宠物档案

宠物档案是推荐系统的核心输入，不是普通资料页。它决定路线、POI、天气风险、社交倾向和宠物分身状态。

P0 最小字段：

| 字段 | 示例 | 用途 |
| --- | --- | --- |
| 名字 | 奶茶 | 个性化表达和宠物分身绑定。 |
| 物种 | 狗、猫 | P0 以狗为主，但保留扩展。 |
| 体型 | 小型、中型、大型 | 判断距离、POI 体型限制、社交风险。 |
| 年龄段 | 幼年、成年、老年 | 判断运动强度和天气风险。 |
| 精力水平 | 1-5 | 判断是否需要放电。 |
| 怕热程度 | 1-5 | 高温风险门控。 |
| 是否怕狗 | 是/否/不确定 | 社区狗密度推荐。 |
| 是否怕车/电动车 | 是/否/不确定 | 路线风险判断。 |
| 社交偏好 | 喜欢社交、适度、避社交 | 路线标签和热力图解释。 |
| 可走时长 | 10/20/40 分钟 | 常走路线推荐。 |
| 主人偏好 | 不打车、不进商场、避主干路 | 方案过滤。 |

档案原则：

- 首次填写应轻量。
- 允许后续通过出行反馈逐步完善。
- 不把医疗诊断写成事实档案。
- 对健康限制使用保守表达，例如“需要更短路线”而不是疾病判断。

## 宠物分身 Agent

宠物分身 Agent 是 Q 版/3D 虚拟宠物模块的产品化定义。它不是装饰，而是宠物画像和路线 Agent 的可视化身体。

### 产品价值

宠物分身承担四个角色：

| 角色 | 说明 |
| --- | --- |
| 宠物画像的可视化身体 | 表达怕热、怕狗、精力、路线偏好等。 |
| 出门前决策助手 | 用状态、表情和短句表达路线是否适合。 |
| 路线记忆入口 | 让复杂数据变成“今天更适合走短线”这样的直觉反馈。 |
| 社区贡献激励 | 上传 POI、反馈路线、标记风险点后触发表情、徽章或成长。 |

### 生成链路

推荐生成链路：

```text
真实宠物照片
-> 宠物主体裁剪/背景去除/特征抽取
-> image-to-image 生成 Q 版 2D 形象
-> 生成多视角 Q 版参考图
-> image/multiview-to-3D 生成低面数模型
-> 自动/半自动 quadruped rig
-> idle / walk / run / sit / sniff 动画
-> GLB/USDZ 进入地图与互动场景
```

可选供应商或技术路线：

| 环节 | 方向 |
| --- | --- |
| 2D Q 版形象 | OpenAI 图像生成、Gemini 图像生成、Stable Diffusion/Flux 私有部署。 |
| 2D 到 3D | Tripo、Meshy、Rodin/Hyper3D、Stable Fast 3D、TripoSR。 |
| 模型优化 | 低面数、1K/2K 贴图、GLB/USDZ、移动端降级。 |
| 动画与 rig | 四足动物 rig、idle/walk/sit/sniff 状态机。 |

### MVP 范围

P0 不要求完整高质量 3D 互动，但必须定义降级路径：

- 用户上传 3 张宠物照片。
- 系统生成 4 个 Q 版 2D 候选。
- 用户选择一个候选作为宠物分身。
- 可选异步生成静态或轻 idle 3D GLB。
- 地图上宠物分身至少表达 3 类反馈：天气风险、路线强度、遇狗概率或 POI 状态。
- 3D 生成失败时降级为 2D 头像，不阻塞地图使用。

### 风险边界

| 风险 | 处理 |
| --- | --- |
| Q 版不像真实宠物 | 让用户确认耳型、毛色、斑纹、尾巴、体型、项圈。 |
| 单张照片无法知道背面 | P0 允许 AI 补多视角，P1 鼓励上传 3-5 张照片。 |
| 四足 rig 不稳定 | 默认 2D 降级，不阻塞核心流程。 |
| 移动端性能 | GLB 目标小于 12MB，低端机自动降级 2D。 |
| 生成成本 | 设置重试次数和月度额度。 |
| 宠物照片隐私 | 原图加密、默认不公开、用户删除宠物时删除衍生资产。 |

## 宠物友好地图与 Dog Need Fulfillment Graph

现有 Dog Need Fulfillment Graph 应升级为宠物友好地图的数据核心。它不是普通 POI 目录，而是记录“哪些空间、路线、时间、入口、风险、服务和社区反馈能帮助这只宠物完成本次出门目标”的结构化图谱。

### 节点类型

| 节点类型 | 示例 |
| --- | --- |
| 居住节点 | 家、小区楼栋、小区门、地库口。 |
| 路线节点 | 常走路线、短程排便线、河边路线、公园路线。 |
| 空间节点 | 草地、广场、阴影路、室内宠物空间、商圈可停留区。 |
| POI 节点 | 宠物友好咖啡店、宠物店、医院、洗护、寄养、水碗。 |
| 风险节点 | 主干路、电动车高压段、狗群聚集点、禁入点、施工点、无遮阴硬质路面。 |
| 交通节点 | 上车点、停车点、电梯、入口、撤离点。 |

### 边与信号

| 类型 | 示例 |
| --- | --- |
| 可步行到达 | 从小区东门到河边短线。 |
| 可完成需求 | 可排便、可嗅闻、可放电、可降压、可社交、可避社交。 |
| 存在风险 | 狗多、电动车多、地面烫、无遮阴、禁入。 |
| POI 规则 | 可进室内、仅户外、需牵引、体型限制。 |
| 社区趋势 | 工作日晚间狗多、周末上午安静、雨后泥泞。 |
| 可信度 | 最近验证时间、多人确认、争议状态。 |

## 常走路线记忆

常走路线是新 PRD 的核心。产品应承认：用户日常遛狗大多数时候不是探索，而是从熟悉路线中选择今天更合适的一条。

### 识别方式

P0 支持两种方式：

1. 用户手动保存路线。
2. 系统根据 3 次以上相似轨迹建议合并为常走路线。

### 路线标签

常走路线应被结构化为以下标签：

- 短程排便线。
- 安静路线。
- 嗅闻路线。
- 社交路线。
- 避狗路线。
- 夜间路线。
- 雨后不推荐路线。
- 高温短走路线。
- 顺路补给路线。

### 路线卡片字段

| 字段 | 说明 |
| --- | --- |
| 路线名称 | 用户命名或系统建议。 |
| 起终点隐私区域 | 不展示精确家庭地址。 |
| 常走时段 | 例如工作日晚 19:00。 |
| 平均时长和距离 | 只用于个人，不直接公开。 |
| 适合需求 | 排便、嗅闻、放松、社交、放电。 |
| 风险标签 | 狗多、车多、无遮阴、雨后泥、禁入。 |
| 社区交叉信号 | 该时段狗多概率、安静程度、热门 POI。 |
| 最近反馈 | 用户和社区的结构化观察。 |

## 出门 Agent 推荐

Agent 是产品决策核心。它不只是聊天，也不是把用户输入改写成地图搜索词，而是完成：

```text
理解目标 -> 召回路线/POI -> 读取宠物画像 -> 读取天气与社区热力 -> 风险门控 -> 排序 -> 解释 -> 反馈写入
```

### 模式 A：无目标日常出门

用户打开产品，只是准备带宠物出门。

输入：

```text
准备带奶茶出门。
```

Agent 需要：

- 读取常走路线。
- 读取当前天气和时间。
- 读取宠物档案。
- 读取社区热力和风险点。
- 判断今天哪条熟悉路线更合适。
- 必要时建议缩短、改时段、换路线或只做楼下微出门。

输出示例：

```text
今天 19:00 不太建议走“东门环线”完整路线：
- 当前天气闷热，连续暴露段偏长；
- 社区热力显示 18:30-19:30 东门草地狗密度偏高；
- 奶茶档案显示怕大狗。

更建议走“河边短线”，控制在 18 分钟内。它更安静，阴影更多，但水碗点仍待验证，建议带水。
```

### 模式 B：有目标附近出行

用户有明确目标，例如咖啡店、宠物医院、宠物店、草地、公园或顺路补给。

输入：

```text
今晚想带狗去能坐下的地方，最好顺路买点狗粮，别遇到太多大狗。
```

Agent 需要：

- 召回宠物友好 POI。
- 过滤宠物规则不匹配的地点。
- 召回路线和交通方式。
- 结合社区狗密度、天气、POI 可信度排序。
- 输出首推、备选和不推荐项。

输出必须包含：

- 推荐地点和路线。
- 推荐理由。
- 宠物友好规则。
- 最近验证时间。
- 狗多概率或社区趋势。
- 风险点和规避建议。
- 备选方案。
- 一键反馈或纠错入口。

### 推荐分

P0 使用规则 + 加权排序，不做复杂黑盒模型。

```text
推荐分 =
目标匹配
+ 宠物适配
+ 天气安全
+ 路线熟悉度
+ POI 可信度
+ 社交匹配
- 风险惩罚
- 绕路成本
```

安全风险必须 hard veto，不能被总分抵消。

触发 hard veto 的情况：

- 高温高湿 + 怕热宠物 + 连续暴露超过阈值。
- 反应犬 + 狗密度高路线。
- 用户明确不接受打车，但方案依赖接驳。
- POI 宠物规则和宠物体型/行为不匹配。
- 当前会话反馈某节点不可用。
- 时间预算不足以完成目标需求。

## 社区热力图

社区热力图用于回答“附近趋势”，不是用于追踪其他用户。

### 展示内容

| 图层 | 说明 |
| --- | --- |
| 狗友常走路线热度 | 脱敏聚合后的路线使用趋势。 |
| 狗多时段 | 某区域在粗时间段内狗密度较高。 |
| 安静路线 | 社区反馈狗少、人少、车少的路线。 |
| 社交路线 | 狗友更常聚集、适合社交的路线。 |
| 风险热力 | 禁入、施工、电动车压力、流浪狗、地面烫等。 |
| 热门 POI | 被确认次数多、最近仍可用的宠物友好地点。 |

### 禁止内容

社区热力图不得展示：

- 单个用户实时位置。
- 单条原始轨迹。
- 可推断家庭地址的起终点。
- 某只具体宠物的行动轨迹。
- 追踪某个狗友现在在哪里的能力。

## UGC POI

UGC POI 是宠物友好地图的供给来源。用户上传的 POI 不是内容流，而是进入 Agent 推荐系统的结构化数据。

### POI 类型

| 类型 | 示例 |
| --- | --- |
| 宠物友好餐饮 | 咖啡店、餐厅、露台。 |
| 活动空间 | 草地、公园、广场、河边步道。 |
| 补给 | 水碗、宠物店、宠物用品柜。 |
| 医疗 | 宠物医院、夜间急诊。 |
| 服务 | 洗护、训练、寄养。 |
| 风险 | 禁入点、狗多点、施工、电动车高压段、流浪狗点。 |
| 交通辅助 | 安全上车点、停车点、入口、电梯。 |

### POI 状态

- 待验证。
- 已验证。
- 有争议。
- 已过期。
- 已隐藏。

### POI 可信度

```text
POI 可信度 =
上传者信誉
+ 多人确认
+ 最近验证时间
+ 照片/签到/商家证据
+ Agent 外部核验
- 纠错次数
- 信息过期
- 争议记录
```

### 上传流程

```text
用户长按地图 / 拍照上传
-> 选择 POI 类型
-> 填写宠物友好规则或风险描述
-> 上传证据
-> 进入待验证状态
-> 附近用户确认或纠错
-> Agent 定期复核
-> 进入推荐系统
```

高风险 POI，例如医院、急救、禁入和危险点，不能只靠单个用户上传进入高置信推荐。

# Agent 架构

## 架构原则

MVP 不建议一开始做真正的多 Agent 系统。建议采用“单一 Orchestrator Agent + 多个确定性能力模块”的架构。

```{mermaid}
flowchart LR
    U[用户] --> UI[产品界面]
    UI --> O[Orchestrator Agent]
    O --> P[宠物档案与宠物分身状态]
    O --> R[常走路线记忆]
    O --> M[宠物友好地图与 POI]
    O --> H[社区热力检索]
    O --> W[天气与风险评估]
    O --> S[推荐排序与解释]
    O --> F[反馈写入]
    O --> V[隐私策略检查]
    P --> S
    R --> S
    M --> S
    H --> S
    W --> S
    V --> H
    S --> UI
    F --> R
    F --> M
    F --> H
```

关键原则：

- 大模型负责理解、综合、解释和生成自然语言。
- 风险门控、隐私策略、POI 可信度和推荐分应尽量结构化、可测试。
- Agent 不得凭空编造 POI、路线或宠物友好规则。
- 对低置信信息必须标注“待验证”或“不确定”。
- 高风险推荐必须保守。

## P1/P2 多 Agent 演进

当数据和工具复杂度上升后，可拆分为：

| Agent | 职责 |
| --- | --- |
| 地图 Agent | 管理 POI、路线、风险点和图谱检索。 |
| 推荐 Agent | 生成有目标和无目标推荐。 |
| 隐私 Agent | 检查路线贡献、热力展示和起终点遮罩。 |
| 商家核验 Agent | 核验宠物友好规则、营业时间、商家认领。 |
| 宠物分身 Agent | 管理形象生成、状态表达和反馈引导。 |

# 数据模型

## 核心对象

| 对象 | 说明 |
| --- | --- |
| `UserProfile` | 居住区域粗粒度、出行偏好、隐私设置、是否参与社区热力。 |
| `PetProfile` | 体型、年龄、品种、精力、怕热/怕冷、怕狗、社交偏好、健康限制。 |
| `PetAvatar` | Q 版图、3D 模型、动画、生成状态、版本、删除状态。 |

| `POIObservation` | 用户上传、签到、纠错、照片、评论、Agent 验证记录。 |

## 事件采集

P0 应采集以下事件：

- 打开出门页。
- 开始路线。
- 结束路线。
- 偏航。
- 停留。
- 收藏路线。
- 拒绝推荐。
- 选择推荐。
- 上传 POI。
- 纠错 POI。
- 标记遇到狗。
- 标记危险点。
- 天气触发提醒。
- 推荐后反馈。
- 宠物分身生成、选择、失败、删除。

事件采集必须遵循最小化原则，并受到隐私设置控制。

# 隐私与安全

## 路线隐私

路线数据是高度敏感数据，可以推断家庭住址、工作地点、日常规律和宠物活动习惯。PRD 必须明确：

- 原始 GPS 轨迹只用于个人路线回放和个性化推荐。
- 社区热力只使用聚合后的格网数据。
- 用户必须明确 opt-in 后才贡献社区热力。
- 起点和终点必须做隐私遮罩，尤其是家、公司、小区门口。
- 用户可以暂停记录、删除历史、退出社区贡献。
- 不展示实时个人位置。
- 不展示单条原始路线。

## 社区热力脱敏策略

| 策略 | 说明 |
| --- | --- |
| 默认 opt-in | 用户明确同意后才贡献社区热力。 |
| 原始 GPS 私有 | 原始轨迹不进入公共展示层。 |
| 聚合格网 | 使用 H3/geohash 等空间格网。 |
| 起终点遮罩 | 家、公司、小区门口附近做隐私半径。 |
| 延迟展示 | 至少延迟 30-60 分钟进入社区热力。 |
| k 匿名阈值 | 达到足够不同用户或 session 后才展示，例如 10 个用户以上。 |
| 粗时间粒度 | 使用“工作日晚上”“周末上午”等，不精确到分钟。 |
| 低密度隐藏 | 低密度区域不展示或加噪声。 |
| 用户控制 | 可暂停贡献、删除历史、退出热力图。 |

## 推荐安全边界

系统不得承诺：

- POI 永远可进。
- 路线绝对安全。
- 一定不会遇到其他狗。
- 医疗和行为判断的确定结论。
- 社区热力等于实时情况。

所有推荐都应展示置信度、最近验证时间、风险和备选。

# 非功能需求

## 性能

| 项目 | 要求 |
| --- | --- |
| 出门推荐 | P95 小于 5 秒返回首屏建议。 |
| 地图 POI 加载 | P95 小于 2 秒加载当前视野基础 POI。 |
| Q 版形象生成 | P95 小于 60 秒。 |
| 3D 模型生成 | 可异步，P95 小于 10 分钟，不阻塞地图使用。 |
| 宠物分身渲染 | 移动端目标 45fps 以上，低端机降级 2D。 |
| 3D 资产大小 | 单个 GLB 目标小于 12MB，贴图默认 1K。 |

## 可用性

- 用户首次出门推荐应在 3 分钟内完成设置。
- 出行后反馈不超过 2 个核心问题。
- POI 上传可以 30 秒内完成基础提交。
- 隐私设置必须在首次路线记录前明确展示。

## 可解释性

每个推荐必须包含：

- 推荐理由。
- 风险因素。
- 社区数据或 POI 证据。
- 备选方案。
- 不确定信息标注。

# 黑客松 / MVP 演示流程

为了让路演稳定，MVP 应准备一个固定演示区域和固定宠物档案。

## 演示输入

| 输入项 | 示例 |
| --- | --- |
| 城市片区 | 上海某社区样例区。 |
| 天气 | 32°C，湿度 70%，工作日晚上。 |
| 宠物 | 柯基，3 岁，中型，怕热 4/5，怕大狗。 |
| 常走路线 | 东门环线、河边短线、便利店补给线。 |
| 社区热力 | 东门草地 18:30-19:30 狗多，河边路线更安静。 |
| 用户目标 | 无目标日常出门，或想顺路买狗粮。 |

## 演示步骤

1. 用户选择宠物档案。
2. 展示 Q 版宠物分身。
3. 用户点击“准备出门”。
4. Agent 读取天气、常走路线、社区热力和 POI。
5. 系统不推荐东门环线完整路线，解释狗多和天气风险。
6. 系统推荐河边短线，提示带水。
7. 用户切换到有目标模式：“顺路买狗粮”。
8. Agent 推荐便利店补给线，展示 POI 可信度和路线风险。
9. 用户走完后反馈“路上狗不多，水碗可用”。
10. 地图更新 POI 和路线信号。

演示要让观众看到：

- 它不是普通路线导航。
- 它会记住常走路线。
- 它会用社区趋势做判断。
- 它会解释为什么推荐或不推荐。
- 它会把用户反馈沉淀到地图。
- 宠物分身不是装饰，而是推荐解释和反馈入口。

# 成功指标

## 核心指标

| 指标 | 说明 |
| --- | --- |
| 出门建议采纳率 | 用户是否真的按推荐出门。 |
| 推荐后完成出门率 | 用户查看建议后是否开始路线记录。 |
| 7 日 / 30 日重复使用率 | 是否形成出门前打开习惯。 |
| 常走路线保存率 | 用户是否愿意让产品记住路线。 |
| 常走路线识别准确率 | 用户是否认可系统识别的 routine。 |
| 推荐解释满意度 | 用户是否理解为什么推荐。 |
| POI 贡献率 | 每 100 个活跃用户贡献多少有效 POI。 |
| POI 验证率 | UGC 信息被其他用户确认的比例。 |
| POI 纠错率 | 信息过期或错误比例。 |
| 路线反馈率 | 出行后是否愿意反馈。 |
| 热力图 opt-in 率 | 用户是否信任社区贡献。 |
| 风险提醒命中率 | 天气、狗多、禁入、施工提醒是否有用。 |
| 宠物分身生成完成率 | 用户是否完成 Q 版或 3D 生成。 |
| 宠物分身使用影响 | 宠物分身是否提升反馈率、留存和贡献率。 |

## P0 验收标准

- 用户可以创建宠物档案，并用档案影响推荐。
- 用户可以保存或识别至少 2 条常走路线。
- 无目标出门时，Agent 能判断今天哪条常走路线更合适。
- 有目标出门时，Agent 能结合 POI、天气、宠物档案和路线风险输出推荐。
- 用户可以上传 POI，并进入待验证状态。
- 出行后反馈能影响下一次推荐。
- 用户可明确控制路线记录和社区热力贡献。
- Q 版宠物分身可以生成并在推荐页表达至少 3 类状态。

# 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 路线数据泄露家庭地址 | 高 | opt-in、起终点遮罩、聚合展示、删除权。 |
| POI 信息过期 | 高 | 最近验证时间、可信度、过期降权、争议标记。 |
| 社区热力冷启动 | 中 | 先做小区域 pilot，使用模拟和核心用户贡献。 |
| 推荐错误导致危险路线 | 高 | hard veto、风险解释、备选方案、保守阈值。 |
| 3D 宠物生成质量不稳定 | 中 | 2D 降级、异步生成、用户确认特征。 |
| 功能发散 | 高 | MVP 聚焦常走路线、出门 Agent、基础 POI 和轻量反馈。 |
| 用户不愿反馈 | 中 | 宠物分身引导、反馈问题不超过 2 个、反馈带来可见价值。 |
| 社区滥用或骚扰 | 高 | 不显示实时位置、不做附近实时找人、举报和审核机制。 |

# 里程碑

## 7 天原型

- 第 1 天：定义样例区域、宠物档案和 3 条常走路线。
- 第 2 天：构建基础 POI 和风险点数据。
- 第 3 天：实现无目标出门推荐和有目标推荐的规则链路。
- 第 4 天：实现 Q 版宠物分身生成或静态 mock。
- 第 5 天：实现 POI 上传和出行后轻量反馈。
- 第 6 天：实现社区热力模拟图层和隐私说明。
- 第 7 天：跑 5 个用户脚本，验证推荐解释是否可理解。

## 4 周 MVP

- 第 1 周：宠物档案、路线记录、常走路线保存。
- 第 2 周：基础 POI、UGC 上传、可信度字段。
- 第 3 周：出门 Agent、天气风险、推荐解释、反馈写入。
- 第 4 周：Q 版宠物分身、热力图 pilot、隐私控制、内测指标。

# 开放问题

- P0 是否优先支持狗，还是也允许猫用户建立档案但不进入路线推荐主链路？
- 常走路线的识别是纯客户端、本地优先，还是服务端聚合？
- 社区热力 opt-in 是否默认关闭，还是在用户完成首次路线后引导开启？
- Q 版/3D 宠物分身是否作为 MVP 路演亮点，还是完全放到 P1？
- POI 的商家认领是否需要提前设计接口，还是 P2 再做？
- 是否需要引入宠物医院、夜间急诊和 SOS，作为后续高情绪场景扩展？
