# 产品定位与机会

## 01. 屏幕里的生活越来越满

你或许有过这样的时刻：拿起手机，解锁，手指悬在屏幕上，忽然想不起来自己原本要做什么。于是顺势点开一个应用，刷几条视频，看几条消息，再问 AI 一个问题。十几分钟过去，信息像潮水一样涌来，心里却留下某种轻飘飘的空白。

这几乎是当代生活的一个缩影。我们拥有越来越多工具，路线可以规划，内容可以生成，服务可以下单，答案可以被即时返回。技术原本承诺替人省下时间，可省下来的时间很快又被新的信息填满。人看似更有效率，身体却常常停在原地；知道的事情越来越多，真正亲身接触的世界越来越少。

看理想那篇文章里提到的“失重感”，放在这里很贴切。很多本该由身体经历的事，被屏幕预览、评分、推荐和生成内容替代了。我们可以收藏很多目的地，却很久没有走进一片草地；可以看见别人拍下的黄昏，却很少注意自己楼下那条路在傍晚是什么颜色。生活被连接得很满，和真实世界的接触却变薄了。

AI 的出现让这种变化更明显。它可以替人整理信息、生成计划、给出安慰，也可以把一个模糊问题迅速变成一个看似完整的答案。可它替不了人感受风从手臂上掠过，替不了人判断一块地面是否烫脚，替不了牵引绳那头突然收紧的一瞬间。越是在什么都能被预测、生成、代劳的时代，身体在场的能力越显得稀缺。

## 02. 宠物把人带回附近

宠物恰好会把人从屏幕里拉出来。狗每天都要出门，主人也就每天都要重新面对脚下那片具体的生活：今天热不热，地面烫不烫，哪条路有阴影，哪片草地能停留，哪个路口电动车多，哪家店门口有水碗，楼下最近总能遇到几只大狗。

这些问题很小，小到很少出现在宏大的城市叙事里。可它们决定了一次出门是放松还是紧张，是一次普通散步，还是一次需要不断安抚和绕行的小型冒险。宠物让城市显出另一种纹理。平时只是“步行 8 分钟”的路，到了遛狗时会被拆成风、声音、气味、地面、牵引绳和狗的反应。

项飙谈“附近的消失”，说的是人对身边世界的感知、叙述和介入能力正在变弱。放在养宠生活里，这个问题会变得非常具体。用户知道城市里哪里热门，却未必知道楼下哪条路今晚适合遛狗；点评里有很多评论，却未必说清这家店今天能不能让狗坐下；地图能给出最短路线，却很难告诉主人这条路对一只怕热、怕车、怕大狗的狗是否合适。

于是，“附近”变成一种生活范围。主人和宠物每天反复经过、反复确认、反复形成经验：路线、地点、天气、人的态度，还有宠物自己的脾气和身体限制，都在里面。

## 03. 当附近变得不好判断

现在的问题并不在于信息太少。普通地图知道道路，本地生活平台知道评分，宠物社区知道经验，运动工具知道轨迹，智能硬件知道位置。真正困难的是，这些信息彼此分散，很少被放回一次具体出门里。

同一家咖啡店，对一只小型犬和一只反应犬，体验可能完全不同；同一条常走路线，在高温、雨后、施工、狗多时段和宠物状态不稳时，也会变成另一条路线；同一条用户评论，如果缺少最近验证、宠物类型、可进入范围和风险上下文，很难变成真正可用的建议。

城市养宠人真正缺少的，是一种把附近重新读懂的能力：在出门前得到一句足够具体的判断，今晚带这只狗出门，附近哪一种选择最稳？

这个判断需要同时理解宠物、路线、地点和时间。它要知道这只狗怕热，就应该避开长时间暴晒；它要知道这只狗怕大狗，就应该降低狗多草地的优先级；它要知道某个 POI 最近没人验证，就应该诚实地标出不确定；它还要记得主人最常走的路线，因为熟悉本身也是一种安全感。

## 04. 爪边的机会

爪边想成为这样一个日常工具：在主人准备出门的那几分钟，把宠物档案、常走路线、天气环境、社区信号、UGC POI 和宠物分身放在同一个判断里，帮助主人决定今天该不该出门、走哪条熟路、是否适合探索新地点、路上可能遇到什么，以及哪些宠物友好地点在此刻真实可用。

普通地图已经很擅长给出路线。爪边要处理的是另一层问题：这条路、这个时段、这个地点，对这只宠物是否合适。

地图在这里承担了更重的工作。它承载用户反复走过的路线、社区留下的观察、地点规则的变化，以及宠物自身的差异。宠物分身也有更具体的职责：把那些差异带回出门前的解释里，告诉主人这只狗怕热、怕车、体力弱、喜欢社交，或者今天不太适合走远。复杂的数据，最后应该变成一句主人听得懂、也愿意采纳的话。

过去，这些判断大多依赖主人的经验。哪条路安静，哪家店能进，哪个时间段狗多，很多答案都藏在一次次散步、邻里闲聊和微信群记录里。这些经验真实，却分散；有温度，却很难被下一次出门调用。

现在，条件正在变得成熟。宠物档案可以记录个体差异，路线记录可以识别日常习惯，UGC 可以把水碗、草地、禁入、施工和店铺规则结构化，社区热力可以提供趋势，Agent 可以把这些信号合成解释。Strava 让个人路线和社区热力变得可见，Waze 证明了用户上报能够改变地图，大众点评和宠物社区积累了大量地点经验。爪边要把这些机制收回到一个更贴近日常生活的问题里：

```text
熟悉路线
  + 宠物个体差异
  + 天气和时间
  + POI 规则
  + 社区反馈
  + Agent 解释
= 一次更稳的附近判断
```

如果 P0 能让用户在出门前得到一句可信、具体、能反馈的建议，产品的价值就会超过一个宠物友好地点列表。它会成为一个帮助养宠人重新读懂附近的日常工具。

# 相关产品调研与可迁移机制

截至 2026 年 6 月，和爪边相邻的产品大致分成四类：宠物友好地图、运动路线产品、UGC 地图、社区产品。它们各自解决了一部分问题，也各自留下了空白。

宠物友好地图回答“哪里可以带宠物”，运动路线产品回答“我走过哪里、大家常去哪里”，UGC 地图回答“现实世界的变化如何进入地图”，社区产品回答“附近经验如何被发现、相信和传播”。爪边要做的，是把这些能力重新翻译成一次具体出门前的判断：现在，带这只宠物，去哪里更稳。

| 产品类型 | 代表产品 | 已解决的问题 | 可迁移机制 | 需要克制的部分 |
| --- | --- | --- | --- | --- |
| 宠物友好地图 | [BringFido](https://www.bringfido.com/)、[DogPack](https://www.dogpackapp.com/)、[Sniffspot](https://www.sniffspot.com/)、Dog Furiendly、DogTrekker | 带宠旅行或本地出门前，不知道哪里能住、吃、玩、看病、托管，地点规则是否可靠。 | 结构化 POI、狗主评论、照片证据、最近验证、宠物规则字段、敏感犬筛选、区域精选。 | 纯目录、泛社区、实时附近狗友、过早进入预约交易。 |
| 运动路线产品 | [Strava](https://support.strava.com/hc/en-us/articles/216918387-Routes-on-Web)、[AllTrails](https://support.alltrails.com/hc/en-us/articles/37227964040852-How-to-use-filters-to-find-trails)、[Komoot](https://support.komoot.com/hc/en-us/articles/10194639751450-Highlights) | 记录个人路线，发现社区常走路线，按活动类型、距离、难度、路面、繁忙程度筛选。 | 常走路线记忆、个人热力图、社区热力、路线卡、宠物语义筛选、地点与路段亮点。 | 排行榜、速度竞争、打卡挑战、把“热门”默认等同于“推荐”。 |

## 宠物友好地图：从“可带宠”到“此刻适合吗”

海外宠物友好地图已经很成熟。[BringFido](https://www.bringfido.com/) 覆盖酒店、餐厅、活动、服务等带宠场景，支持用户提交地点、评论和照片；[DogPack](https://www.dogpackapp.com/) 把地图、Feed、公园群聊、AI 搜索和宠物服务放在一起；[Sniffspot](https://www.sniffspot.com/) 则把公共狗公园的不确定性拆开，提供按小时预约的私人放风空间，尤其强调围栏、无其他狗可见或可听、敏感犬友好等条件。

这些产品证明了一件事：宠物友好不能只是一枚标签。对用户真正有用的信息，是地点规则和使用条件。咖啡店能不能带狗，只是第一层；狗能不能进室内、是否只能坐露台、是否限制体型、是否需要牵引、是否有水碗、最近有没有被验证，才决定一次出门是否顺利。

爪边可以迁移的能力：

1. `POI 规则卡`：可进室内 / 仅露台 / 只能门口等待、体型限制、牵引要求、水碗、阴影、草地、围栏、地面材质、宠物服务、最近验证时间。
2. `可信度体系`：官方或商家认领、用户最近验证、多用户一致、照片证据、过期未验证、规则存在分歧。
3. `宠物适配标签`：怕热、怕狗、怕车、老年犬、幼犬、短鼻犬、反应犬，分别影响地点和路线排序。
4. `区域精选`：早期先做 1-3 个养宠高密度片区，把可信地点、常走路线、狗多时段和风险点做厚。
5. `高不确定场景兜底`：急诊医院、跨城住宿、规则不明商场等场景，可以设计“需电话确认”或“人工确认中”的状态。

需要克制的是“目录越大越好”的冲动。GoPetFriendly 这类老牌带宠旅行目录的式微说明，纯地点列表容易被通用搜索、地图和 AI 问答压缩。爪边的价值要落在更细的日常判断里：地点、路线、天气、时间、宠物个体差异和最近社区反馈共同给出答案。

## 运动路线产品：从运动成绩到宠物适配

[Strava](https://support.strava.com/hc/en-us/articles/216918387-Routes-on-Web) 把个人轨迹、社区热力、路线规划、照片和运动成就连在一起；[AllTrails](https://support.alltrails.com/hc/en-us/articles/37227964040852-How-to-use-filters-to-find-trails) 用难度、长度、海拔、活动类型、dog-friendly、繁忙程度等筛选项降低户外路线的不确定性；[Komoot](https://support.komoot.com/hc/en-us/articles/10194639751450-Highlights) 的 Highlights 把用户实际走过、看过、推荐过的地点和路段沉淀在地图上，成为路线规划的一部分。

这些机制可以迁移到爪边，但评价尺度要换掉。运动产品关心距离、速度、爬升、成绩和探索，宠物出门更关心熟悉、稳定、温度、地面、狗密度、车流、噪声和宠物当天状态。对一只反应犬来说，社区热力越高，可能越需要绕开；对一只老年犬来说，最好的路线也许不是风景最好，而是短、平、阴影多、可随时回家。

爪边可以迁移的能力：

1. `常走路线记忆`：自动识别用户反复走过的路线，沉淀为排便线、安静线、社交线、高温短线、雨后慎走线。
2. `个人热力图`：只给主人自己看，帮助用户理解宠物最近的活动范围、偏好和过度重复的路线。
3. `社区热力图`：脱敏后按时段展示狗多、安静、风险、友好 POI、阴影路线等趋势。
4. `路线卡`：距离、预计耗时、阴影 / 暴晒、地面、草地、水源、狗多时段、复杂路口、最近反馈。
5. `宠物语义筛选`：把 AllTrails 的户外筛选翻译成城市遛狗筛选，如避狗、避车、少台阶、有草地、有阴影、有室内可停留点。

需要避免的是运动化叙事。排行榜、segments、速度、距离挑战、连续打卡都可能诱导用户让宠物过量运动。爪边的路线推荐要服务“更稳”，而不是“更强”。

## UGC 地图：让附近变化进入系统

UGC 地图最值得学习的地方，不是把评论贴到地图上，而是一套信号进入系统的闭环：

```text
结构化采集
-> 证据和时效
-> 多人或权威验证
-> 置信度与过期
-> 风险分级使用
-> 隐私与滥用治理
```

Waze 让用户上报事故、拥堵、封路和路障，并通过路过用户确认；Google Local Guides 让用户贡献评论、照片、问答、地点编辑和事实核查；OpenStreetMap 的 Notes 把“这里可能有问题”先作为待处理观察，而不是直接写入正式地图；Mapillary 用带位置的街景照片为地图编辑提供证据；FixMyStreet 把公共空间问题做成可追踪的报告。

爪边需要把 UGC 分成两类：

| 信息类型 | 示例 | 产品处理方式 |
| --- | --- | --- |
| 短时风险信号 | 狗多、未牵引犬、施工、地面烫、积水、烟花噪声、临时禁入、流浪犬 | 影响短时间推荐，30 分钟到数小时自动衰减，附近路过用户轻点“还在 / 不在 / 不确定”。 |
| 稳定 POI / 规则 | 店铺可进范围、体型限制、水碗、围栏、草地、常见狗密度、路线风险点 | 进入地点或路线卡，但必须带来源、最近验证、置信度和争议状态。 |

爪边可以迁移的能力：

1. `贡献类型分层`：POI、规则、临时风险、路线反馈、照片证据、纠错、最近验证分开建模。
2. `Claim / Verified / Needs recheck`：单个用户观察先是 Claim，多人确认或官方确认后才进入 Verified，过期或争议后回到 Needs recheck。
3. `极轻反馈`：走完路线只问 1-2 个问题，例如“狗多吗”“是否顺利”“有没有新风险”。
4. `照片证据脱敏`：默认模糊人脸、儿童、车牌、门牌、宠物铭牌电话，保留事实，不保留可识别人。
5. `贡献信誉`：依据纠错准确率、被确认率、争议率、近 90 天活跃和本地常走区域贡献计算，不做公开积分榜。

这部分尤其要谨慎。宠物路线会暴露住址、作息和宠物身份，风险比普通地点评论更高。爪边不展示个人实时位置，不展示单条原始轨迹；社区热力要延迟、聚合、k 匿名，并默认隐藏起终点附近区域。

## 社区产品：把经验沉淀为地图信号

小红书让普通生活经验变成可搜索、可收藏、可传播的内容；Nextdoor 把邻里新闻、安全提醒、本地推荐、活动和问答组织在地理社区里；Reddit 和豆瓣小组则证明，围绕一个兴趣或身份的长期讨论，可以沉淀出规则、经验和社区文化。宠物社区、BarkHappy、DogPack 这类产品又进一步说明，狗主确实愿意分享地点、照片、活动和走失提醒。

但爪边的社区不宜变成一个新的内容流。用户出门前需要的是判断，不是再刷十分钟经验帖。社区对爪边的价值，应该体现在地图和推荐系统里：谁最近确认过这个水碗还在，哪条路晚上狗多，哪个门口新贴了禁宠标识，哪片草地雨后容易积水。

爪边可以迁移的能力：

1. `结构化贡献`：上传水碗、草地、禁入、施工、狗多、吵闹、电动车多、店铺可进入范围，每条贡献都有类型、时间、证据和适配对象。
2. `附近问答`：用户问“今晚 8 点附近哪里适合怕大狗的小型犬”，Agent 汇总路线、天气、POI、社区反馈后回答，并展示理由和不确定性。
3. `片区社区`：先围绕少数高密度养狗片区运营，沉淀常走路线、狗多时段、宠物友好店、风险点和可信贡献者。
4. `风险上报规则`：危险犬、疑似投毒、可疑人、冲突等内容只描述地点状态和客观行为，不点名具体人、狗、住址、车牌。
5. `轻社交`：可以有感谢、纠错、补充证据和共同验证，不做陌生人私信、实时狗友匹配和公开附近狗主列表。

社区产品能带来供给，也会带来治理压力。P0 阶段，爪边应该让社区成为地图判断的证据层，而不是让地图成为社区内容的入口。

## 对爪边的产品建议

从调研看，爪边最应该迁移的是“路线与地点如何被验证、记住、解释”的机制。P0 可以优先落到以下产品能力：

| 优先级 | 能力 | 说明 |
| --- | --- | --- |
| P0 | POI 可信度卡 | 显示可进入范围、体型限制、牵引要求、最近验证、证据来源、是否存在分歧。 |
| P0 | 常走路线卡 | 把用户熟悉路线沉淀为可复用对象，记录狗密度、阴影、地面、路口、水源、最近反馈。 |
| P0 | 出门判断解释 | Agent 给出“今天推荐走哪条 / 不建议去哪”的同时，说明天气、宠物状态、路线和社区信号。 |
| P0 | 走后极轻反馈 | 只问少量问题，把“狗多、顺利、新风险、POI 变化”写回系统。 |
| P1 | 社区热力图 | 按时段展示狗多、安静、风险、友好 POI 和路线趋势，默认脱敏和延迟。 |
| P1 | 贡献信誉与纠错 | 贡献者不靠积分排名，而靠被确认率、纠错命中率和本地贡献质量影响权重。 |
| P2 | 商家 / 场地确认 | 支持宠物友好商家认领、规则更新；可探索私人放风空间或宠物活动预约。 |

最终要形成的产品感不是“这里有很多宠物友好地点”，而是“我准备出门时，爪边知道我、知道这只宠物，也知道今天附近发生了什么”。这会让地图从静态目录变成一个可被日常经验持续修正的附近系统。

# 用户场景与 P0 范围

## 用户分层

| 用户类型 | 核心诉求 | P0 是否覆盖 | 备注 |
| --- | --- | --- | --- |
| 城市日常遛狗人 | 每天从熟悉路线中选一条更合适的。 | 是 | P0 主用户。 |
| 敏感犬或反应犬主人 | 避开狗、人流、电动车、儿童、噪声和复杂路口。 | 是 | 直接验证宠物差异化推荐。 |
| 老年犬、幼犬、短鼻犬或慢病犬主人 | 更保守的天气、距离和强度控制。 | 是 | 不做医疗诊断，只做出门风险提醒。 |
| 新手养狗人 | 不知道什么路线、地点和时段更安全。 | 是 | 依赖解释和保守建议。 |
| 新搬家宠物家庭 | 缺少附近宠物友好地点、风险点和狗友路线认知。 | 是 | 适合地图和社区信号。 |
| 地图贡献型狗友 | 愿意上传 POI、纠错、标记狗多和风险点。 | 是 | 是供给冷启动关键用户。 |
| 宠物友好商家和服务方 | 想让宠物主人知道规则和服务。 | 否 | P2 再做商家端和认领。 |
| 猫或其他宠物主人 | 需要宠物友好地点和出行建议。 | 部分 | P0 先以狗为主，档案字段保留扩展。 |

## 核心使用场景

| 场景 | 用户问题 | 产品回答 |
| --- | --- | --- |
| 无目标日常出门 | 准备带狗下楼，今天走哪条常走路线？ | 结合宠物档案、天气、社区信号和历史反馈，推荐熟悉路线或降级路线。 |
| 有目标附近出行 | 想去咖啡店、宠物店、医院、公园或草地。 | 召回 POI 和路线，判断宠物适配度、路线风险和 POI 可信度。 |
| 高风险天气出门 | 高温、降雨、湿度高、地面烫、空气质量差。 | 判断是否适合出门，建议缩短、改时段、换路线或去室内。 |
| 避狗或社交 | 用户想避开狗，或想让宠物适度社交。 | 根据社区信号和宠物画像推荐安静路线或社交路线。 |
| 地图贡献 | 用户发现水碗、草地、禁入、狗多或宠物友好店。 | 结构化上传 POI 或风险点，进入待验证和推荐系统。 |
| 宠物分身生成 | 用户上传宠物照片，希望生成 Q 版/3D 形象。 | 生成宠物分身，并让它参与路线解释、反馈和社区身份。 |

## P0

这一版先不追求完整地图，也不追求复杂社区。用户打开爪边时，产品至少要完成一件事：根据这只宠物、今天的天气、熟悉路线和附近地点，给出一个可以马上采纳的出门建议；走完之后，把新的经验记回来。

| 功能组 | P0 范围 | 验收信号 |
| --- | --- | --- |
| 宠物档案 | 体型、年龄、精力、怕热、怕狗、怕车、可走时长、社交偏好。 | 推荐结果能体现宠物差异。 |
| 常走路线 | 用户手动保存或系统自动识别 2-5 条 routine。 | 用户打开产品时能看到“今天常走路线是否适合”。 |
| 出门 Agent | 支持无目标常走路线判断和有目标推荐。 | Agent 能输出推荐理由、风险和备选。 |
| 宠物友好地图 | 基础 POI、风险点、水源、草地、公园、医院、宠物店。 | POI 能进入推荐，地图展示只是基础能力。 |
| UGC POI | 上传、纠错、待验证、最近验证时间。 | 新 POI 进入待验证状态，并可被低置信引用。 |
| 路线反馈 | 走完后 1-2 个轻量问题，例如狗多吗、是否顺利、是否有风险。 | 反馈能影响下一次推荐或 POI 可信度。 |
| 天气与风险 | 温度、降雨、时间段、连续暴露、狗密度等基础风险。 | 高风险时能拒绝或降级路线。 |
| 宠物分身基础版 | Q 版 2D 头像；可异步生成静态或轻 idle 3D。 | 宠物分身能表达天气、路线强度、遇狗概率等状态。 |
| 隐私控制 | 路线记录开关、社区贡献 opt-in、起终点隐藏。 | 用户能明确控制是否贡献社区热力。 |

## 后续路线图与暂不进入范围

P1/P2 保留为产品路线图，不进入 P0 版本。

| 阶段 | 方向 | 目标 |
| --- | --- | --- |
| P1 产品化 | 社区热力图展示、个人路线交叉分析、POI 可信度体系、主动提醒、弱社交、互动版宠物分身。 | 让真实用户可以反复使用，并形成社区数据飞轮。 |
| P2 平台化 | 商家端、服务交易、硬件接入、高级宠物分身、多 Agent、社区治理。 | 支撑多城市、多服务方、商业化和复杂协作。 |

暂不进入 P0：

- 泛社区信息流。
- 实时看到其他狗或狗主的位置。
- 排行榜和竞技化路线挑战。
- 完整商家后台。
- 真实交易、支付和服务履约。
- 完整 AR 宠物互动。
- 医疗诊断和行为训练诊断。
- 全国地图数据铺开。
- 与硬件深度绑定。

# 核心体验闭环

## 首次使用

首次使用要让用户在 3 分钟内进入第一次出门推荐，不应让用户填大量资料。

```{mermaid}
flowchart LR
    A[打开产品] --> B[创建宠物档案]
    B --> C{是否生成宠物分身}
    C -- 是 --> D[上传宠物照片并生成 Q 版头像]
    C -- 否 --> E[使用默认头像]
    D --> F[选择是否记录路线]
    E --> F
    F --> G[设置隐私与社区贡献]
    G --> H[保存第一条常走路线或开始出门]
```

最低完成项：

- 创建宠物档案。
- 选择是否生成 Q 版宠物分身。
- 选择是否记录路线。
- 明确社区热力贡献默认关闭或明确 opt-in。

## 出门推荐：无目标与有目标

出门推荐是产品的主体验。它分为无目标日常出门和有目标附近出行，但底层都遵循同一个判断链路：

```{mermaid}
sequenceDiagram
    participant U as 用户
    participant UI as 出门页
    participant A as Orchestrator Agent
    participant R as 常走路线与 POI
    participant C as 宠物档案、天气、社区信号
    participant G as 风险门控
    participant P as 宠物分身与反馈

    U->>UI: 打开出门页或输入目标
    UI->>A: 发送当前位置、目标和隐私设置
    alt 无明确目标
        A->>R: 读取常走路线和历史反馈
    else 有明确目标
        A->>R: 解析目标并召回 POI / 路线
    end
    A->>C: 读取宠物画像、天气和社区趋势
    A->>G: 提交候选方案和上下文
    alt 触发 hard veto
        G-->>A: 返回拒绝原因和可降级方案
        A-->>UI: 给出缩短、改时段或替代地点
    else 未触发 hard veto
        G-->>A: 返回可推荐候选
        A-->>UI: 排序并生成推荐解释
    end
    UI->>P: 同步路线强度、天气风险和遇狗概率
    P-->>UI: 用宠物状态辅助解释
    U->>UI: 选择执行、换一个或拒绝
    UI->>A: 写入选择与出行结果
    A->>R: 更新路线记忆、POI 可信度和社区趋势
```

推荐页必须让用户一眼看懂：

- 推荐走哪条路线或去哪个 POI。
- 为什么推荐。
- 为什么不推荐某些常走路线。
- 有哪些风险和备选。
- 信息的置信度和最近验证时间。
- 走完后如何反馈。

## 地图贡献与反馈沉淀

UGC POI 和路线反馈会以结构化数据进入 Agent 推荐系统，避免退化成普通社区内容流。

```{mermaid}
flowchart LR
    A[用户发现地点或风险] --> B[长按地图或拍照上传]
    B --> C[选择 POI / 风险类型]
    C --> D[填写规则、证据和适合宠物类型]
    D --> E{风险等级}
    E -- 普通 POI --> F[进入待验证]
    E -- 高风险点 --> G[提高核验门槛]
    F --> H[附近用户确认或纠错]
    G --> H
    H --> I[更新可信度和最近验证]
    I --> J[进入 Agent 推荐系统]
    J --> K[影响推荐、备选和风险提示]
```

高风险 POI，例如医院、急救、禁入和危险点，不能只靠单个用户上传进入高置信推荐。

# 核心能力设计

## 宠物档案

宠物档案是推荐系统的核心输入，承担的职责超过普通资料页。它决定路线、POI、天气风险、社交倾向和宠物分身状态。

| 字段 | 示例 | 用途 |
| --- | --- | --- |
| 名字 | 奶茶 | 个性化表达和宠物分身绑定。 |
| 物种 | 狗、猫 | P0 以狗为主，但保留扩展。 |
| 体型 | 小型、中型、大型 | 判断距离、POI 体型限制、社交风险。 |
| 年龄段 | 幼年、成年、老年 | 判断运动强度和天气风险。 |
| 精力水平 | 1-5 | 判断是否需要放电。 |
| 怕热程度 | 1-5 | 高温风险门控。 |
| 是否怕狗 | 是/否/不确定 | 社区狗密度推荐。 |
| 是否怕车/电动车 | 是/否/不确定 | 路线风险判断。 |
| 社交偏好 | 喜欢社交、适度、避社交 | 路线标签和热力解释。 |
| 可走时长 | 10/20/40 分钟 | 常走路线推荐。 |
| 主人偏好 | 不打车、不进商场、避主干路 | 方案过滤。 |

档案原则：

- 首次填写应轻量。
- 允许后续通过出行反馈逐步完善。
- 不把医疗诊断写成事实档案。
- 对健康限制使用保守表达，例如“需要更短路线”，避免写成疾病判断。

## 常走路线记忆

常走路线是产品差异化核心。产品应承认：用户日常遛狗大多数时候都在熟悉路线中选择今天更合适的一条，探索新地点只占少数。

| 规格 | P0 要求 |
| --- | --- |
| 识别方式 | 用户手动保存路线，或系统根据 3 次以上相似轨迹建议合并为常走路线。 |
| 路线标签 | 短程排便线、安静路线、嗅闻路线、社交路线、避狗路线、夜间路线、雨后不推荐路线、高温短走路线、顺路补给路线。 |
| 路线卡片字段 | 路线名称、起终点隐私区域、常走时段、平均时长和距离、适合需求、风险标签、社区交叉信号、最近反馈。 |
| 反馈写入 | 出行后反馈可更新狗多、车多、无遮阴、泥泞、禁入、水源、POI 可用性等信号。 |
| 隐私约束 | 原始路线默认私有，社区展示只能使用聚合后的趋势。 |

## 出门 Agent 推荐

Agent 的职责不止于聊天，也不止于把用户输入改写成地图搜索词。它要完成：

```text
理解目标
-> 召回路线 / POI
-> 读取宠物画像
-> 读取天气与社区信号
-> 风险门控
-> 排序
-> 解释
-> 反馈写入
```

| 模式 | 输入 | Agent 动作 | 输出 |
| --- | --- | --- | --- |
| 无目标日常出门 | “准备带奶茶出门。” | 读取常走路线、天气、宠物档案、社区信号和风险点。 | 推荐最合适的熟悉路线，必要时建议缩短、改时段或只做楼下微出门。 |
| 有目标附近出行 | “今晚想带狗去能坐下的地方，顺路买狗粮，别遇到太多大狗。” | 召回 POI 和路线，过滤规则不匹配地点，结合狗密度、天气、可信度排序。 | 输出首推、备选、不推荐项、风险点、规避建议和反馈入口。 |

P0 使用规则 + 加权排序，不做复杂黑盒模型。

```text
推荐分 =
目标匹配
+ 宠物适配
+ 天气安全
+ 路线熟悉度
+ POI 可信度
+ 社交匹配
- 风险惩罚
- 绕路成本
```

安全风险必须 hard veto，不能被总分抵消。触发 hard veto 的情况包括：

- 高温高湿 + 怕热宠物 + 连续暴露超过阈值。
- 反应犬 + 狗密度高路线。
- 用户明确不接受打车，但方案依赖接驳。
- POI 宠物规则和宠物体型/行为不匹配。
- 当前会话反馈某节点不可用。
- 时间预算不足以完成目标需求。

## 地图 POI、风险点与社区信号

宠物友好地图承担结构化数据层的职责，记录哪些空间、路线、时间、入口、风险、服务和社区反馈能帮助这只宠物完成本次出门目标。`Dog Need Fulfillment Graph` 可以作为内部架构概念，但主 PRD 应优先使用更清晰的对象表达。

| 对象 | 示例 | 作用 |
| --- | --- | --- |
| 路线 | 常走路线、短程排便线、河边路线、公园路线。 | 支撑日常出门推荐。 |
| 空间 | 草地、广场、阴影路、室内宠物空间、商圈可停留区。 | 判断宠物需求能否完成。 |
| POI | 宠物友好咖啡店、宠物店、医院、洗护、水碗。 | 支撑有目标附近出行。 |
| 风险点 | 主干路、电动车高压段、狗群聚集点、禁入点、施工点、无遮阴硬质路面。 | 支撑风险门控和路线降级。 |
| 社区信号 | 狗多时段、安静路线、风险趋势、热门 POI、最近验证。 | 支撑概率判断，不追踪个人。 |
| 规则 | 可进室内、仅户外、需牵引、体型限制、最近验证时间。 | 判断 POI 是否适合本次出行。 |

UGC POI 必须结构化：

| 维度 | 内容 |
| --- | --- |
| POI 类型 | 宠物友好餐饮、活动空间、补给、医疗、服务、风险、交通辅助。 |
| POI 状态 | 待验证、已验证、有争议、已过期、已隐藏。 |
| 可信度因子 | 上传者信誉、多人确认、最近验证时间、照片/签到/商家证据、Agent 外部核验、纠错次数、过期和争议记录。 |
| 推荐使用 | 低置信 POI 可以被引用，但必须标注待验证；高风险 POI 需要更高确认门槛。 |

社区信号在 P0/P1 的边界：

| 阶段 | 做法 |
| --- | --- |
| P0 | 使用小区域 pilot、模拟数据或低粒度聚合信号作为推荐输入。 |
| P1 | 做面向用户展示的社区热力图，展示狗友常走趋势、狗多时段、安静路线、风险热力和热门 POI。 |
| 禁止 | 展示单个用户实时位置、单条原始轨迹、可推断家庭地址的起终点、某只具体宠物行动轨迹、追踪某个狗友现在在哪里。 |

## 宠物分身与状态表达

宠物分身不应发展成第二条产品主线。它的职责是把宠物画像、路线判断、风险提示和反馈入口变成一个更直觉的身体界面。

| 角色 | 说明 |
| --- | --- |
| 宠物画像的可视化身体 | 表达怕热、怕狗、精力、路线偏好等。 |
| 出门前决策助手 | 用状态、表情和短句表达路线是否适合。 |
| 路线记忆入口 | 让复杂数据变成“今天更适合走短线”这样的直觉反馈。 |
| 社区贡献激励 | 上传 POI、反馈路线、标记风险点后触发表情、徽章或成长。 |

### 生成链路

2D Q 版形象不作为主要技术难点。P0 统一使用 [GPT Image 2](https://platform.openai.com/docs/guides/image-generation) 生成和编辑干净的 Q 版主设定图，不在首版比较多套 image-to-image 模型。照片输入主要服务于“像不像这只宠物”，3D 之后的资产质量、骨骼绑定、动作映射和自然语言交互才是重点。

```{mermaid}
flowchart LR
    A[宠物照片] --> B[GPT Image 2 生成 Q 版候选]
    B --> C[用户选择主设定图]
    C --> D[多视角参考图]
    D --> E[Image / Multi-view to 3D]
    E --> F[GLB / USDZ 资产包]
    F --> G[骨骼绑定与动作状态机]
    G --> H[PetCue 自然语言交互]
    H --> I[地图推荐页表达状态]
```

P0 范围：

- 用户上传 1-3 张宠物照片；若用户愿意补拍，优先引导上传正面、侧面、背面。
- 系统使用 GPT Image 2 生成 4-8 张 Q 版 2D 候选，用户选择一张作为主设定。
- 主设定图用于生成多视角参考；侧面和背面缺失时，系统要标注“背面花纹和尾巴细节可能是推测”。
- 3D 生成异步执行，不阻塞地图使用；首版只承诺静态展示或轻 idle。
- 宠物分身至少表达 3 类状态：天气风险、路线强度、遇狗概率或 POI 状态。
- 3D 资产生成、绑定或移动端 QA 失败时，降级为 2D 头像、多角度卡片或轻量 turntable。

标准资产包：

| 文件 | 用途 |
| --- | --- |
| `avatar.png` | GPT Image 2 生成的 Q 版主设定图。 |
| `model.glb` | Android/Web/Three.js 主资产。 |
| `model.usdz` | iOS AR Quick Look 和系统预览。 |
| `preview_front/right/back/left.png` | 质量审核和用户确认。 |
| `metadata.json` | 模型来源、输入图 hash、生成参数、面数、贴图尺寸、状态、失败原因。 |
| `quality_report.json` | 四肢、尾巴、耳朵、花纹、文件大小、贴图、移动端帧率等检查结果。 |

### 3D 资产与骨骼绑定

真正的难点不在“生成一个 3D 模型”，而在它能否像一只可信的宠物分身那样稳定出现、移动、表达状态。很多 image-to-3D 输出可以看，但拓扑、四肢、尾巴、贴图和关节未必适合动画。

P0 资产验收：

| 项目 | P0 标准 |
| --- | --- |
| 文件格式 | GLB 为主，USDZ 作为 iOS 预览和 AR 输出。 |
| 模型大小 | 单个 GLB 目标小于 10-12MB。 |
| 面数 | 移动端优先 10k-50k triangles，必要时提供 LOD。 |
| 贴图 | 1-2 张 1K/2K 贴图；Q 版优先 baseColor + normal，复杂 PBR 延后。 |
| 材质 | 移动端可用 toon / unlit 版本，避免贴图烘焙阴影过重。 |
| QA | 四肢和尾巴完整、地面接触合理、法线/UV 无明显问题、iOS/Android 可预览。 |

骨骼绑定分阶段处理：

| 阶段 | 做法 | 目标 |
| --- | --- | --- |
| P0 | 静态模型 + 轻 idle；必要时用预制动作或简单骨骼。 | 让宠物进入地图和推荐页，不让动画质量拖住主链路。 |
| P1 | 做 quadruped rig、retopo、skinning、idle/walk/sit/sniff/pant 等动作。 | 让宠物状态表达更自然，并支持基础互动。 |
| P2 | 针对狗/猫体型建立专用 rig 模板和动作库，支持尾巴、耳朵、嘴型和表情 morph target。 | 做长期可扩展的宠物分身系统。 |

最小动作集：

```text
idle / thinking / happy / cautious / hot / tired / scared_dog
wag_tail / pant / sniff / point_route / nod / shake_head
ask_feedback / celebrate_feedback
```

### 自然语言交互

宠物分身不直接理解开放自然语言。用户说的话先进入 Orchestrator Agent，经过路线、天气、POI、社区信号、风险门控和排序后，再把结构化结果映射成宠物动作。

```text
用户自然语言
-> Intent Router
-> Orchestrator Agent
-> 路线 / 天气 / POI / 社区信号 / 宠物档案工具
-> RiskGate 与排序
-> Explanation Builder
-> PetCue Mapper
-> 3D Runtime 播放动作、表情、气泡和反馈入口
```

客户端只接收受控的 `PetCue`，避免模型临场编造动作：

```json
{
  "schema": "pet_cue.v1",
  "trace_id": "rec_123",
  "assistant_text": "今晚走楼下短线更稳，河边大狗多。",
  "pet_state": {
    "mood": "cautious",
    "energy": "low",
    "risk": "dog_density",
    "confidence": 0.78
  },
  "actions": [
    { "type": "animation", "name": "cautious_idle", "priority": 60 },
    { "type": "map_point", "target": "route_A" },
    { "type": "feedback_prompt", "question": "回来后帮我确认一下狗多不多？" }
  ]
}
```

状态映射：

| 推荐信号 | 宠物表现 |
| --- | --- |
| 高温 / 地面热 | pant、低能量、担心表情、建议短线。 |
| 路线适合 | wag、跳一下、指向路线。 |
| 遇狗概率高 | 后退、警觉、耳朵收起、建议避开。 |
| POI 低置信 | 歪头、问号、提示需要确认。 |
| 出行后反馈 | 坐下看用户，递出 1-2 个反馈按钮。 |

P0 不做实时语音宠物。文本交互 + 宠物状态表达更可控，也更容易验证推荐是否可信。P1 可加入 TTS 或按住说话；P2 再考虑 Realtime speech-to-speech、lip-sync、打断和长期记忆。

# Agent 架构与数据

## Agent + Workflow 架构

P0 不做真正的多 Agent 系统。爪边需要一个会理解、会调用工具、会解释结果的出门编排层。多个 Agent 自由讨论路线是否安全，会让首版推荐变得更难测试和复盘。

更合适的做法是：保留一个面向用户的 Orchestrator Agent，把路线、POI、社区信号、隐私、风险和排序都拆成可测试的确定性模块。Agent 负责把用户的模糊目标变成结构化请求，再把工具返回的结果讲清楚。路线安全、POI 可信度、隐私边界和 hard veto 不能交给提示词临场判断。

```{mermaid}
flowchart LR
    U[用户] --> UI[出门页]
    UI --> V[隐私与授权网关]
    V --> O[Orchestrator Agent]
    O --> I[意图路由与上下文整理]
    I --> P[宠物档案工具]
    I --> R[RouteTool 与 MapMatchTool]
    I --> M[POI 与风险检索]
    I --> C[社区信号与可信度]
    I --> W[天气与环境工具]
    P --> G[RiskGate 风险门控]
    R --> G
    M --> G
    C --> G
    W --> G
    G --> S[排序与解释生成]
    S --> O2[输出隐私检查]
    O2 --> UI
    UI --> F[反馈与 Observation 写入]
    F --> V
    V --> T[可信度与状态机]
    T --> C
    T --> M
    T --> R
```

一次推荐的最小链路：

```text
用户输入 / 出门事件
-> 隐私与授权检查
-> 意图路由
-> 候选路线、POI、风险点和社区信号召回
-> 风险门控
-> 排序与解释
-> 输出隐私检查
-> 用户选择与出行反馈
-> Observation 写入
-> 路线记忆、POI 状态和社区趋势更新
```

模块职责：

| 模块 | P0 职责 | 不做什么 |
| --- | --- | --- |
| Orchestrator Agent | 理解用户目标，整理宠物、天气、路线和 POI 上下文，调用工具，生成可读解释。 | 不直接计算路线，不凭空判断 POI 规则，不绕过风险门控。 |
| Intent Router | 区分日常出门、有目标出行、POI 上传、纠错、风险上报、出行反馈。 | 不承担复杂推荐。 |
| RouteTool / MapMatchTool | 召回常走路线、计算候选路线、做 GPS 轨迹吸附、输出距离和时长。 | 不判断这条路线是否适合宠物。 |
| POISearchTool | 根据位置、路线 corridor、类别和宠物条件召回 POI。 | 不把单条 UGC 当成事实。 |
| CommunitySignalTool | 读取 H3/geohash 聚合后的狗密度、风险趋势、热门 POI 和最近验证。 | 不返回单个用户轨迹或实时位置。 |
| TrustState Service | 把上传、纠错、签到、照片和最近验证聚合为 POI 状态、风险状态和可信度。 | 不让单次贡献直接覆盖主表。 |
| Privacy Gateway | 做授权检查、字段最小化、起终点遮罩、空间降精度、k 匿名和低密度隐藏。 | 不让推荐链路直接访问完整家庭轨迹。 |
| RiskGate | 处理高温、反应犬、禁入、危险点、低置信高风险 POI 等 hard veto。 | 不允许总分抵消安全风险。 |
| Ranker | 对通过 RiskGate 的候选做排序，输出分项分数和不推荐理由。 | 不处理安全 veto。 |
| Explanation Builder | 把结构化结果转为推荐页文案和宠物分身状态。 | 不制造没有证据的安慰性解释。 |

这一层最重要的并非 Agent 数量。每次推荐都应能回放：输入快照、召回候选、风险 veto、排序分、最终解释和用户反馈都应被记录，便于测试、复盘和纠错。

## 首版技术架构

首版面向 iOS 和 Android，移动端建议使用 React Native + Expo Development Build。Expo Go 只用于早期试验；地图、定位、后台任务、推送、深链和 3D 资产需要尽早在自有 development build 中验证。

代码仓库建议采用轻量 monorepo。这个结构服务于协作和一致性，让客户端、API、Agent 和数据任务共享同一套核心对象、推荐契约和隐私规则，不提前铺平台化大系统。

| 层 | 建议 |
| --- | --- |
| `apps/mobile` | iOS / Android 主应用，承载出门页、地图、路线记录、POI 上传和宠物分身展示。 |
| `services/api` | BFF/API，负责账号、档案、路线、POI 和推荐请求。 |
| `services/agent` | Orchestrator Agent、意图路由、工具调用、推荐解释和 trace 记录。 |
| `services/jobs` | 社区热力聚合、POI 过期检查、宠物分身生成等异步任务。 |
| `packages/domain` | 共享 `PetProfile`、`RouteSession`、`POI`、`RecommendationContext` 等核心类型。 |
| `packages/recommendation` | 推荐分、hard veto、POI 可信度、隐私遮罩和输出解释规则。 |
| `packages/geo` | 路线、地图匹配、H3/geohash 聚合、route buffer 和空间查询。 |
| 数据层 | Postgres/PostGIS 存路线、POI、Observation 和热力格网；对象存储保存照片、2D 分身和 3D 资产。 |

工程边界：

- 客户端负责体验、采集和展示。
- 服务端负责推荐、脱敏、可信度、热力聚合、风险门控和生成任务。
- 3D 分身异步生成，不阻塞地图和推荐。
- P0 不拆复杂多 Agent，也不为后台运营台铺大系统。

## UGC 证据层

爪边的社区 UGC 应作为推荐系统的证据层，而不是新的内容流。每条贡献先成为带时间、地点、来源、证据、风险等级和状态的 Observation，经过去重、脱敏、可信度计算、争议处理和过期策略后，才聚合为 POI、路线和风险信号。

```{mermaid}
flowchart LR
    A[UGC 输入] --> B[结构化 Observation]
    B --> C[隐私与安全清洗]
    C --> D[实体匹配 POI RouteSegment RiskPoint]
    D --> E[可信度与状态机]
    E --> F[社区信号索引]
    F --> G[推荐召回]
    G --> H[RiskGate 与排序]
    H --> I[Agent 生成解释]
    I --> J[出行后反馈写回]
    J --> B
```

Observation 状态机：

```text
Claim
-> Pending Review
-> Verified
-> Disputed
-> Needs Recheck
-> Expired / Hidden
```

迁移到产品里的规则：

- POI 上传先进入 `Claim`，可被低置信引用，但不能当成确定事实。
- 纠错不会直接覆盖主表，会生成新的 `CorrectionObservation`。
- `Verified` 超过 TTL 后降级为 `Needs Recheck`，过期信息只能作为提示，不能作为强推荐依据。
- 出现冲突时进入 `Disputed`，推荐层降权，并展示“信息有分歧”。
- 宠物医院、急救、禁入、疑似投毒、危险犬、流浪犬聚集、施工危险等高风险字段，需要多人最近确认、官方/商家来源或人工审核。
- 贡献信誉用于可信度加权和审核优先级，不做公开排行榜。

## 多 Agent 演进边界

P0 采用单 Orchestrator。P1/P2 数据和工具复杂度上升后，可以把后台能力拆成专门 Agent，但它们应服务于 workflow，不能替代确定性模块。

| 阶段 | 架构形态 | 适合拆出的能力 |
| --- | --- | --- |
| P0 | 单 Orchestrator Agent + 确定性工具层。 | 意图理解、工具调用、解释生成。 |
| P1 | 显式 workflow / state graph + 异步任务。 | POI 证据刷新、路线记忆更新、社区反馈处理、低置信信息复核。 |
| P2 | 多 Agent 后台协作。 | 地图 Agent、推荐 Agent、隐私/合规 Agent、商家核验 Agent、社区治理 Agent、宠物分身 Agent。 |

即使进入 P2，隐私和风险仍然是强制网关。推荐 Agent 只能消费可授权、可聚合、可回放的位置特征；隐私模块决定什么数据能离开端侧、能进入社区统计、能被模型使用。

| UGC 可信度 | [OpenStreetMap Notes / Changesets](https://wiki.openstreetmap.org/wiki/Notes)、[StreetComplete](https://github.com/streetcomplete/StreetComplete)、[MapRoulette](https://github.com/maproulette/maproulette3) | 地图问题提示、现场小任务、changeset、review queue、微任务复核。 | 用于设计 POI 上传、最近验证、纠错、复核和争议机制。 |
| 社区治理 | [Ushahidi](https://github.com/ushahidi/platform)、[FixMyStreet](https://github.com/mysociety/fixmystreet)、[Discourse](https://github.com/discourse/discourse) | 多来源上报、公开状态流转、责任路由、信任等级、举报和审核队列。 | 用于高风险上报、社区治理和内容审核。 |
| 隐私与脱敏 | [OpenDP](https://github.com/opendp/opendp)、[ARX](https://github.com/arx-deidentifier/arx)、[diffprivlib](https://github.com/IBM/differential-privacy-library) | 差分隐私、k-anonymity、重识别风险分析、匿名化质量评估。 | P0 先做最小化、遮罩和 k 匿名；P1 再引入 DP 噪声和隐私风险回归。 |

P0 的建议技术组合可以保持朴素：OSM 数据 + PostGIS 空间库 + H3 聚合 + OSRM 或 GraphHopper 路线/轨迹匹配 + MapLibre 展示 + 一个 Orchestrator Agent。框架名字本身不构成差异，差异来自宠物画像、常走路线、社区证据和风险门控能否被稳定地串起来。

## 核心对象

| 对象 | 说明 |
| --- | --- |
| `UserProfile` | 居住区域粗粒度、出行偏好、隐私设置、是否参与社区热力。 |

| `AvatarAsset` | GLB、USDZ、预览图、贴图、面数、文件大小、LOD、质量报告。 |
| `AvatarRig` | 骨骼、skinning、动作 clip、morph target、绑定状态和适配物种。 |
| `PetCue` | Agent 输出给客户端的受控动作契约，包含文案、状态、表情、动画、地图指向和反馈问题。 |
| `RouteSession` | 一次出门记录，含时间、天气、路线 polyline、停留点、距离、反馈。 |
| `RoutineRoute` | 系统识别或用户保存的常走路线。 |

## 事件采集

P0 应采集以下事件：

- 打开出门页。
- 开始路线。
- 结束路线。
- 偏航。
- 停留。
- 收藏路线。
- 拒绝推荐。
- 选择推荐。
- 上传 POI。
- 纠错 POI。
- 标记遇到狗。
- 标记危险点。
- 天气触发提醒。
- 推荐后反馈。
- 推荐请求、候选召回、风险 veto、排序结果和解释曝光。
- 隐私授权、撤回、遮罩和社区热力 opt-in / opt-out。
- POI 状态变更、争议、过期和复核。
- 宠物分身生成、选择、失败、删除。
- Q 版候选生成、用户选择、重新生成和删除。
- 3D 资产生成、供应商、候选选择、QA 失败、绑定状态和降级原因。
- `PetCue` 下发、动作播放、用户点击反馈和渲染失败。

事件采集必须遵循最小化原则，并受到隐私设置控制。

# 可信边界与风险

## 位置与路线隐私

路线数据是高度敏感数据，可以推断家庭住址、工作地点、日常规律和宠物活动习惯。PRD 必须明确：

- 原始 GPS 轨迹只用于个人路线回放和个性化推荐。
- 社区热力只使用聚合后的格网数据。
- 用户必须明确 opt-in 后才贡献社区热力。
- 起点和终点必须做隐私遮罩，尤其是家、公司、小区门口。
- 用户可以暂停记录、删除历史、退出社区贡献。
- 不展示实时个人位置。
- 不展示单条原始路线。

## 社区热力脱敏

| 策略 | 说明 |
| --- | --- |
| 默认 opt-in | 用户明确同意后才贡献社区热力。 |
| 原始 GPS 私有 | 原始轨迹不进入公共展示层。 |
| 聚合格网 | 使用 H3/geohash 等空间格网。 |
| 起终点遮罩 | 家、公司、小区门口附近做隐私半径。 |
| 延迟展示 | 至少延迟 30-60 分钟进入社区热力。 |
| k 匿名阈值 | 达到足够不同用户或 session 后才展示，例如 10 个用户以上。 |
| 粗时间粒度 | 使用“工作日晚上”“周末上午”等，不精确到分钟。 |
| 低密度隐藏 | 低密度区域不展示或加噪声。 |
| 用户控制 | 可暂停贡献、删除历史、退出热力图。 |

## 推荐安全与可解释性

系统不得承诺：

- POI 永远可进。
- 路线绝对安全。
- 一定不会遇到其他狗。
- 医疗和行为判断的确定结论。
- 社区热力等于实时情况。

每个推荐必须包含：

- 推荐理由。
- 风险因素。
- 社区数据或 POI 证据。
- 备选方案。
- 不确定信息标注。

## 非功能需求

| 类别 | P0 要求 | 验收口径 |
| --- | --- | --- |
| 出门推荐 | P95 小于 5 秒返回首屏建议。 | 用户点击“准备出门”后能快速看到可执行建议。 |
| 地图 POI 加载 | P95 小于 2 秒加载当前视野基础 POI。 | 地图不因 POI 加载阻塞主要操作。 |
| Q 版形象生成 | GPT Image 2 生成 P95 小于 60 秒。 | 用户能看到 4-8 个干净候选，并选择主设定图。 |
| 3D 模型生成 | 可异步，P95 小于 10 分钟。 | 失败时降级为 2D，不阻塞地图使用。 |
| 3D 资产 QA | 自动输出预览、面数、贴图、文件大小和移动端预览结果。 | 不合格资产不能进入默认展示。 |
| 骨骼绑定 | P0 可静态或轻 idle；P1 再要求完整 quadruped rig。 | 绑定失败时仍可使用静态分身。 |
| 宠物分身渲染 | 移动端目标 45fps 以上，低端机降级 2D。 | 动画不影响地图交互。 |
| 3D 资产大小 | 单个 GLB 目标小于 12MB，贴图默认 1K。 | 保障移动端加载和缓存。 |
| 首次设置 | 首次出门推荐应在 3 分钟内完成设置。 | 不要求用户一次填完所有档案。 |
| 出行后反馈 | 不超过 2 个核心问题。 | 反馈成本足够低，能持续积累数据。 |
| POI 上传 | 30 秒内完成基础提交。 | 先提交结构化信息，再补充证据。 |

## 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 路线数据泄露家庭地址 | 高 | opt-in、起终点遮罩、聚合展示、删除权。 |
| POI 信息过期 | 高 | 最近验证时间、可信度、过期降权、争议标记。 |
| 社区热力冷启动 | 中 | 先做小区域 pilot，使用模拟和核心用户贡献。 |
| 推荐错误导致危险路线 | 高 | hard veto、风险解释、备选方案、保守阈值。 |
| 3D 宠物生成质量不稳定 | 中 | 2D 降级、多候选生成、用户确认特征、自动 QA。 |
| 四足骨骼绑定质量不稳定 | 中 | P0 只承诺静态或轻 idle，P1 再做 quadruped rig 和动作库。 |
| 功能发散 | 高 | P0 聚焦常走路线、出门 Agent、基础 POI 和轻量反馈。 |
| 用户不愿反馈 | 中 | 宠物分身引导、反馈问题不超过 2 个、反馈带来可见价值。 |
| 社区滥用或骚扰 | 高 | 不显示实时位置、不做附近实时找人、举报和审核机制。 |
