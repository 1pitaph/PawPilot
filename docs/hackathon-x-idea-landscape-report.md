# X/社媒与黑客松创意研究报告

日期：2026-06-13  
范围：不受当前 PRD 限制；不限宠物，也不限地图或路线规划。  
方法：并行调用 3 个子 agent，分别调研 X/社媒上流行的 agent/AI 产品创意、以往黑客松获奖或高质量项目、以及可迁移的创意模式；随后用公开来源做交叉核验。

## 0. 执行摘要

这次研究里最重要的结论是：**好的 hackathon/X 创意，通常不是“找一个新行业 + 套一个 AI”，而是把一个原本需要人费心、费时、费关系、费判断的场景，压缩成一个可演示的“魔法动作”，并且让这个动作在真实生活里能反复发生。**

如果拿宠物方向举例，单纯“遛狗路线规划”确实偏弱。大多数人遛狗走熟路，路线本身不是主要痛点；“模糊目标 -> 推荐路线”只是更自然的入口，但仍然可能停在浅层。真正更有机会的痛点在于：

- **不确定性**：这家店到底能不能带狗？现在人多不多？会不会遇到大狗？草坪能不能进？店员今天会不会拒绝？
- **信任成本**：附近谁能临时帮忙？这个寄养/遛狗员靠不靠谱？点评过期了吗？
- **紧急场景**：宠物走失、生病、被咬、临时出差、极端天气，需要的是行动剧本，不是信息页。
- **附近关系的缺失**：地图给了 POI，却没有给“附近的社会知识”：隐性规则、邻里互助、临时协作、可信关系、实时状态。

所以宠物方向如果要结合项飙“附近的消失”，我建议不要做“宠物友好地图”本身，而是做：

> **一个把附近重新变得可感、可信、可行动的宠物 Agent。**

地图只是底图；Agent 的价值是**问、查、验证、协调、记忆、预警、生成行动材料、必要时发起临时协作**。它不是告诉你“走哪条路”，而是帮你判断“今晚带这只狗出门，附近哪些选择是真的可用、舒服、安全、有人情味，并且如果出问题怎么办”。

从 X/社媒与黑客松案例看，最值得迁移的创意组合是：

1. **Bunkermode 型危机响应层**：多信号监测、分级触发、自动执行、完整留痕。迁移到宠物就是走失/受伤/冲突/临时无人照料的 SOS Agent。
2. **GibberLink 型机器对机器协商**：当对方也是系统或商家后台时，Agent 不必用人类低效语言。迁移到附近服务，就是 Agent 替你询问店铺规则、预约、确认库存、同步宠物约束。
3. **Conclave 型多 Agent 审议**：不同角色互相挑战，最后输出带理由和审计记录的建议。迁移到高风险生活决策，就是“规划 Agent + 安全 Agent + 证据 Agent + 社交 Agent”互相制衡。
4. **TreeHacks/PennApps 型现实世界闭环**：语音、相机、AR、机器人、LoRa、无人机、OCR，把 AI 接到真实空间与真实动作上，而不是停留在聊天。
5. **Anthropic hackathon 型领域专家爆发**：律师、医生、基础设施从业者用 AI 做出过去只有工程团队才能做的产品。机会不在“AI 工具本身”，而在懂问题的人第一次能把流程做成软件。

一句话判断：**宠物赛道里，路线不是核心；附近的可信行动层才是核心。**

## 2. 先回答：什么样的创意会在 X 和黑客松里出彩？

### 2.1 X 上容易传播的创意

X 上传播强的项目，往往具备这些特征：

1. **一句话就能激起想象**
   - “两个 AI 发现对方都是 AI 后，切换成机器语言沟通。”
   - “手绘草图直接变成可运行 UI。”
   - “用自然语言控制你的电脑。”
   - “一个虚拟小镇里，AI 角色自己生活、聊天、形成关系。”

2. **demo 比解释更强**
   - 看一眼视频就能懂，甚至能立刻转发。
   - GibberLink 的传播点不是“音频协议”，而是“AI 和 AI 之间不再需要说人话”这个画面。
   - draw-a-ui / Make Real 的传播点不是代码生成，而是“草图变产品”的瞬间。

3. **它改变用户对界面的想象**
   - AutoGPT/BabyAGI：不是你问一句，它答一句；而是你给目标，它自己拆任务。
   - Open Interpreter：不是聊天框，而是电脑遥控器。
   - AI Town：不是 agent 列表，而是可观察社会。
   - v0/Bolt/Lovable：不是写代码，而是描述产品并发布。

4. **它让“未来”显得很近**
   - X 喜欢的不是完整商业闭环，而是“范式门缝”：用户第一次看到一种新的交互方式。
   - 但真正产品化时，需要从“魔法 moment”回到可复用场景。

### 2.2 黑客松获奖项目常见结构

黑客松评委通常会奖励这些东西：

1. **现实痛点明确**
   - 医疗、急救、灾害、无障碍、教育、隐私、金融、基础设施。

2. **技术杠杆明显**
   - 语音、视觉、AR、机器人、无人机、LoRa、ZK、LLM、多 agent、地图/空间建模。

3. **现场可 demo**
   - 一个输入，一个变化，一个输出。
   - 最好能看到真实世界被改变：机械臂动作、AR 指示、地图亮起、报告生成、电话/短信被辅助。

4. **前后对比强**
   - 原来需要专家，现在普通人也能做。
   - 原来需要 10 步，现在 1 步。
   - 原来信息看不见，现在可视化。
   - 原来人会慌，现在有自动剧本。

5. **风险边界说得清**
   - 高风险领域的好项目通常不宣称“替代医生/律师/急救员”，而是做副驾驶、解释器、分诊、准备材料、升级路径。

### 2.3 一个判断公式

可以用下面公式判断创意是否值得继续：

```text
好创意 = 强痛点 × 强魔法感 × 可重复使用 × 可验证输出 × 可逐步运营
```

反过来，很多看起来聪明但容易失败的创意是：

```text
弱创意 = 新技术 × 弱场景 × 无复用 × 无验证 × 无信任路径
```

“遛狗路线规划”的问题就在这里：它可能有技术与地图，但痛点和复用未必强。用户不是每天都想探索新路线；用户更常需要的是“确定这次出门不会踩坑”。

## 4. 创意模式库：从案例里抽象出的 14 种模式

### 模式 1：一键化代理

**定义**：把原来需要用户跨多个工具完成的流程，压成一个目标输入或一个按钮。

典型案例：Zapier、IFTTT、Calendly、AutoGPT、InsuClaimAI、Nyxie。

适合场景：

- 用户反复做同一套流程。
- 流程步骤多，但每步规则相对清楚。
- 可以从半自动/人工后台开始。

风险：

- 一上来做成万能自动化平台，最后没有一个场景足够深。
- 工具调用失败时，用户不信任系统。

宠物迁移：

- “我下周出差 3 天，帮我安排猫的喂养、钥匙交接、备用医院、支付和每日照片回传。”
- “狗疫苗快到期了，帮我找附近可预约的医院、询价、生成提醒和记录。”

### 模式 2：旧行为的新入口

**定义**：需求早就存在，但旧入口太笨；用语音、相机、位置、短信、AR、LLM 重新做入口。

典型案例：GroupMe、Twitter 早期短信状态更新、Open Interpreter、draw-a-ui。

适合场景：

- 用户本来就会做，只是做起来麻烦。
- 新入口更接近真实意图。

风险：

- 只有入口新，结果没有变好。

宠物迁移：

- 用户不想点筛选项，而是说：“我今晚想带狗出去坐一会儿，不想太吵，最好顺路买药。”
- 这不是路线需求，而是**附近可用性判断**。

### 模式 3：临时协同网络

**定义**：为一个短时事件临时组队、通知、分工、汇报。

典型案例：GroupMe、Meshworks、灾害通信、走失搜寻群。

适合场景：

- 事情有时效性。
- 多人行动比单人行动有效。
- 地理位置重要。

风险：

- 冷启动。
- 消息噪音。
- 误报和骚扰。

宠物迁移：

- 走失宠物 2 小时内创建附近搜寻队。
- 自动生成分区、海报、目击表单、物业/门店/兽医通知清单。
- 用户不只是“发帖”，而是启动一个短期协作系统。

### 模式 4：隐藏信息可视化

**定义**：把看不见、分散、过期、只能靠熟人问的信息，变成可见图层。

典型案例：Waze、NASA Space Apps、SkySplat、Project Naptha、AR 急救。

适合场景：

- 真实世界状态会影响决策。
- 信息在现场可验证。
- 小数据聚合后有大价值。

风险：

- 地图空。
- 信息过期。
- 用户误以为图层是绝对真实。

宠物迁移：

- 不只是“宠物友好店”，而是“今天是否仍允许、是否有水、是否拥挤、是否有大狗、是否有草坪、是否有冲突记录、最近一次验证时间”。

### 模式 5：众包地图

**定义**：每个用户贡献一个很小的报告，聚合后所有人受益。

典型案例：Waze、Ushahidi、FixMyStreet、Space Apps 开放数据项目。

适合场景：

- 数据需要本地密度。
- 用户愿意在完成自己任务后顺手反馈。
- 有审核或信誉机制。

风险：

- 虚假信息。
- 过期信息。
- 贡献者疲劳。

宠物迁移：

- 店铺宠物政策变更。
- 狗公园拥挤程度。
- 最近有无蜱虫、投毒、玻璃碎片、施工噪音。
- 走失宠物目击。

### 模式 6：AI 识别 + 下一步行动

**定义**：识别不是目的，识别后的动作才是价值。

典型案例：Google Lens、Plantix、Project Naptha、Dermalab、PostVisit.ai。

适合场景：

- 输入对象边界清楚。
- 输出能转成下一步行动，而不是只有标签。

风险：

- 误诊式自信。
- 高风险领域没有人工兜底。

宠物迁移：

- 拍宠物皮肤、便便、食品标签、药盒、疫苗本。
- Agent 输出风险分级、要问兽医的问题、附近医院选项、是否需要立即就医。
- 明确不做诊断，而做 triage 和问诊材料准备。

### 模式 7：垂直知识 Copilot

**定义**：把一个资料源可控、规则明确、答案需要引用的领域做成副驾驶。

典型案例：GitHub Copilot、Perplexity、ChatPDF、CrossBeam、EU AI Agent。

适合场景：

- 规则复杂但有文本依据。
- 用户需要解释和行动。

风险：

- 只是聊天套壳。
- 幻觉。
- 无引用。

宠物迁移：

- 当地养宠规定、租房宠物条款、航空/酒店/商场宠物政策、疫苗证明要求。
- Agent 输出“可以/不可以/要准备什么/证据链接/建议措辞”。

### 模式 8：信任层市场

**定义**：先解决“不敢找谁”，再解决“找得到谁”。

典型案例：Airbnb、Rover、Uber、临时服务市场。

适合场景：

- 需求高价值。
- 供给分散。
- 信任、责任、保险、留痕是关键。

风险：

- 双边冷启动。
- 纠纷处理被低估。
- 审核成本高。

宠物迁移：

- 附近可信遛狗员、临时寄养、上门喂养、宠物接送。
- Agent 不只是匹配，而是做筛选、询价、确认、协议、支付、过程记录。

### 模式 9：人工后台先行

**定义**：先用 concierge service 验证需求，后面再自动化。

典型案例：很多 on-demand 服务早期都这样；Anthropic hackathon 的领域专家产品也常是先把自己脑内流程产品化。

适合场景：

- 用户愿意为结果付费。
- 自动化前需要摸清流程。
- 每次人工服务都能沉淀规则。

风险：

- 人工成本隐藏。
- demo 顺畅，真实运营亏损。

宠物迁移：

- “今晚临时找人照看宠物”的 agent，早期可以人工打电话、确认、做表格。
- 真正资产是服务流程、商家响应率、用户约束模板。

### 模式 10：个人数据日志变建议

**定义**：把碎片记录变成趋势、异常和下一步建议。

典型案例：Fitbit、Strava、Mint、Limitless/Rewind、Nyxie。

适合场景：

- 用户关心趋势。
- 数据采集轻。
- 建议能改变行为。

风险：

- 只有图表，没有行动。
- 录入负担过大。

宠物迁移：

- 饮食、体重、散步、排便、用药、情绪、攻击/害怕场景。
- Agent 每周总结：“这周夜间抓挠增加；和换粮时间相关；建议观察 48 小时，并准备这些问题问医生。”

### 模式 11：游戏化现实行动

**定义**：用轻游戏机制促进真实行为，而不是为了积分而积分。

典型案例：Duolingo、Pokemon Go、Strava challenge。

适合场景：

- 行为本身值得重复。
- 奖励强化真实动机。

风险：

- 徽章空洞。
- 为打卡牺牲安全。

宠物迁移：

- 社会化训练任务。
- 胆小狗逐步暴露练习。
- 老年犬温和活动计划。
- 不是“走更多路”，而是“做更合适的训练”。

### 模式 12：开发者抽象

**定义**：把复杂能力包装成 API/SDK/协议，供别人构建。

典型案例：PhoneGap、MCP、Goose、Gitingest、Code2MCP。

适合场景：

- 目标用户是开发者或平台方。
- 痛点是集成复杂。

风险：

- 技术漂亮但没人接。
- 文档和生态维护成本高。

宠物迁移：

- 宠物/附近 SDK：宠物友好地点、规则验证、附近服务、预约、宠物档案、风险图层。
- 更适合 B2B 或平台基础设施，不一定适合第一阶段 C 端。

### 模式 13：SOS 模式

**定义**：高压场景中快速升级、分级触发、通知正确的人、记录证据。

典型案例：Bunkermode、Good Samaritan、Meshworks、Noonlight、what3words、AIxCC。

适合场景：

- 时间敏感。
- 用户容易慌。
- 正确动作路径相对清楚。

风险：

- 误报。
- 责任边界。
- 恐慌营销。

宠物迁移：

- 走失宠物 SOS。
- 被咬/中毒/突发病症应急。
- 极端天气临时安置。
- 自动生成海报、通知附近用户、联系兽医/物业/收容所、收集目击。

### 模式 14：魔法一刻的实用化

**定义**：输入简单、输出立刻可见，并能接真实用途。

典型案例：Shazam、背景移除、Magic Eraser、draw-a-ui、screenshot-to-code、GibberLink。

适合场景：

- 演示 5 秒可懂。
- 输出能拿去用。

风险：

- 只有玩具感，没有第二次使用理由。

宠物迁移：

- 一张宠物照片生成寻宠海报、领养页、宠物名片、店铺询问话术。
- 一张店门口照片识别宠物政策并生成“是否可带狗入内”的询问。

## 5. 从这些模式看：Agent 真正适合解决什么？

### 5.1 不适合 Agent 的场景

不要为了 Agent 而 Agent。下面这些场景往往不适合：

- 规则简单、按钮更快。
- 用户只想看一个静态列表。
- 错误成本高但没有人工兜底。
- 数据无法验证，却要求 Agent 做确定判断。
- 用户需求极低频，且没有情绪强度。

这也是“路线规划”作为核心切入点的问题：地图路线本身已经被高德、百度、Google、Apple Maps 做得很好；用户熟悉路线时也不需要 agent；探索路线时又不一定有高频复用。

### 5.2 适合 Agent 的场景

Agent 适合这些场景：

1. **目标模糊，但约束很多**
   - “想带狗出去坐坐，不想太吵，别遇大狗，顺路买药。”
   - 用户说不清筛选项，但能说出感受和约束。

2. **信息分散且会过期**
   - 店铺政策、天气、拥挤度、用户评价、最近报告、营业时间、库存。

3. **需要替用户去问/确认/协调**
   - 发消息、打电话、预约、询价、确认规则。

4. **需要记住个人偏好与历史**
   - 狗怕大狗、怕电动车、不能上楼梯、喜欢水、主人不喜欢人多。

5. **需要分级风险判断**
   - 不确定是否要去医院。
   - 不确定是否要取消行程。
   - 不确定是否要通知邻居。

6. **需要输出可执行材料**
   - 海报、清单、话术、路线备选、问诊材料、证据包、通知模板。

### 5.3 Agent 的产品价值不是“聊天”，而是“行动闭环”

一个强 Agent 产品通常要有这几层：

```text
用户目标
  -> 意图澄清
  -> 约束建模
  -> 多源信息检索
  -> 证据与置信度
  -> 生成行动方案
  -> 执行/协调/确认
  -> 反馈与记忆
```

只有前四步，很像搜索；做到后四步，才开始像 agent。

## 6. “附近的消失”如何和 Agent 结合？

### 6.1 先把“附近”从地图上救出来

项飙讲“附近的消失”，可以粗略理解为：现代生活让人更容易接入远方的抽象系统，却更难感知身边具体的人、关系、空间规则和生活纹理。

传统地图解决的是：

- 哪里有店？
- 怎么走？
- 多久到？
- 评分多少？

但“附近”真正包含的是：

- 这家店今天的店员是否允许带狗？
- 哪个入口不容易被保安拦？
- 哪块草地其实默认可以玩，哪块最近在管？
- 哪个时段大狗多？
- 谁家有同品种狗可以交流？
- 走失时谁最可能看到？
- 哪个宠物医院夜间真的接诊？
- 邻里之间有什么可求助的微弱关系？

这些不是 POI，而是**附近知识、附近关系、附近状态、附近互助**。

所以，“宠物友好地图”如果只是把宠物友好店铺标出来，确实单薄。更好的命题是：

> 地图负责空间，Agent 负责把空间里的不确定性转成可行动的信任。

### 6.2 Agent 在“附近”里的 6 个角色

#### 角色 1：附近民族志记录员

Agent 不只抓点评，而是把用户反馈、店铺规则、现场观察、照片、对话记录结构化：

- “这家咖啡店允许小型犬进露台，但雨天露台关闭。”
- “周末下午 4-6 点大型犬多，胆小狗不推荐。”
- “保安 A 通常允许牵绳进入，保安 B 会拒绝。”
- “店门口有水碗，但最近一次验证是 40 天前。”

这是“附近”的微知识。

#### 角色 2：实时验证员

Agent 替用户做确认：

- 给店铺发消息：“今晚 8 点能带一只 8kg 柯基坐户外位吗？”
- 给宠物医院确认：“现在是否能接急诊？大概等待多久？”
- 给寄养员确认：“是否接未绝育公犬？是否有单独空间？”

地图告诉你“可能可以”；Agent 帮你确认“这次可以”。

#### 角色 3：约束翻译器

用户说的是模糊目标：

- “别太吵。”
- “不想碰到大狗。”
- “它今天有点紧张。”
- “想顺便买点药。”

Agent 需要翻译成检索条件：

- 噪音、拥挤度、其他狗密度、空间大小、可停留时间、附近药店库存、天气、回家距离、备用方案。

#### 角色 4：临时协调员

当出现事件时，Agent 组织附近协作：

- 走失搜寻。
- 临时寄养。
- 夜间就医接送。
- 宠物聚会/社会化训练。
- 狗公园冲突提醒。

这和 GroupMe、Meshworks 的模式一致：**短时、地理、多人、通知价值高**。

#### 角色 5：信任与留痕层

任何附近互助都会遇到信任问题：

- 谁可以进我家喂猫？
- 谁可以牵走我的狗？
- 临时寄养出问题谁负责？
- 付款、钥匙、照片、时间如何留痕？

Agent 可以做：

- 身份/历史记录。
- 双方约定。
- 照片回传。
- 地理轨迹。
- 保险/押金/支付。
- 纠纷记录。

#### 角色 6：公共空间倡议助手

如果大量用户报告“附近没有宠物友好空间”，Agent 不只是抱怨收集器，还能生成：

- 社区建议书。
- 物业沟通材料。
- 宠物友好设施需求报告。
- 店铺合作清单。
- 小区文明养宠公约草案。

这才真正把“附近”从消费地图变成生活共同体。

### 6.3 产品命题：从“路线”改成“附近可用性”

我建议把核心 job 改成：

> 当养宠人有一个模糊生活目标时，Agent 帮他判断附近哪些选择在“这一次”真实可用、适合他的宠物、风险可控，并能替他完成确认和后续动作。

示例输入：

```text
今晚 8 点想带奶茶出门 40 分钟。它胆小，怕大狗，今天不想走太远。
我想找个能坐一会儿的地方，顺便买驱虫药。
```

理想输出不是一条路线，而是：

- A 方案：最近、风险低，已验证可带狗，药店有库存。
- B 方案：空间更舒服，但今晚人多，不推荐胆小狗。
- C 方案：雨天备选，室内不可进，但门口可等。
- 每个方案给出置信度、证据、最近验证时间、需确认事项。
- Agent 可一键发消息给店铺确认。
- 出门后允许用户反馈：“店员拒绝了”“今天有 3 只大狗”“水碗没了”。

这就是“附近的消失”的反向产品化：**让用户重新拥有附近的具体判断力。**

## 7. 候选方向清单：不限宠物的 24 个可切入创意

下面是从案例里抽象后得到的候选方向。每个方向都可以单独做成产品，也可以迁移到宠物/附近。

### 方向 1：生活版 Bunkermode

**一句话**：当风险事件发生时，Agent 不是提醒你，而是启动个人化应急剧本。

灵感来源：Bunkermode、Good Samaritan、Meshworks、Noonlight。

适用场景：

- 宠物走失/中毒/被咬。
- 老人跌倒/走失。
- 孩子放学异常。
- 租房漏水/停电/火灾。
- 账户被盗/银行卡异常。

MVP demo：

- 输入一个事件：“狗走失，最后出现小区东门，10 分钟前。”
- Agent 自动生成海报、地图分区、通知话术、附近宠物医院/物业/门卫清单、目击表单。
- 每条线索进入时间线。

关键价值：

- 高压场景下，人最缺的是冷静执行。
- Agent 可以把通用应急知识转成用户附近的具体行动。

风险：

- 误报和骚扰。
- 高风险建议责任边界。
- 需要明确“辅助行动，不保证结果”。

### 方向 2：附近可用性 Agent

**一句话**：不是查地点，而是判断“这次去是否真的可用”。

灵感来源：Waze、Project Naptha、NASA Space Apps、项飙“附近”。

适用场景：

- 宠物友好。
- 亲子友好。
- 无障碍出行。
- 老人友好。
- 清真/素食/过敏饮食。
- 夜间自习/办公。

MVP demo：

- 用户输入：“带狗、怕吵、想坐 30 分钟、顺路买药。”
- Agent 输出 3 个方案，带证据、风险、需确认点。
- 一键给店铺发确认消息。

关键价值：

- 大众地图解决“在哪里”，不解决“适不适合我这次去”。

风险：

- 数据冷启动。
- 需要验证时间戳和置信度。

### 方向 3：规则/许可 Copilot

**一句话**：复杂规则不再让用户自己读，Agent 输出可执行结论和材料。

灵感来源：CrossBeam、EU AI Agent、ArcAid。

适用场景：

- 装修许可。
- 租房合同。
- 宠物登记。
- 签证/入境。
- 学校申请。
- 医保/保险理赔。

MVP demo：

- 上传合同或政策链接。
- Agent 回答：“能不能养宠？有哪些限制？需要什么材料？风险条款在哪里？”

关键价值：

- 用户要的是结论、证据和下一步，不是全文解释。

风险：

- 法律责任。
- 必须引用来源，标注不确定性。

### 方向 4：诊后/问诊准备 Agent

**一句话**：把医疗/兽医信息从“听不懂”变成“知道下一步怎么做”。

灵感来源：PostVisit.ai、TheraSpeak、Dermalab。

适用场景：

- 宠物看病后医嘱解释。
- 人类诊后解释。
- 慢病管理。
- 心理咨询后行动计划。

MVP demo：

- 上传医嘱/检查单。
- Agent 输出简单解释、观察清单、复诊提醒、要问医生的问题。

关键价值：

- 高风险领域不一定要替代专家，做解释和准备就有价值。

风险：

- 医疗合规。
- 不应输出诊断式确定结论。

### 方向 5：证据包 Agent

**一句话**：把用户散落的照片、聊天、订单、定位、票据整理成可提交材料。

灵感来源：InsuClaimAI、DataDaddy、GPT Researcher。

适用场景：

- 保险理赔。
- 消费维权。
- 宠物医疗纠纷。
- 租房押金。
- 交通事故。

MVP demo：

- 用户上传 20 张照片和聊天记录。
- Agent 整理时间线、证据清单、缺失材料、提交模板。

关键价值：

- 这类流程烦、低频但痛感强。

风险：

- 隐私。
- 证据真实性与篡改风险。

### 方向 6：领域专家 vibe coding 平台

**一句话**：给律师、医生、教师、物业、兽医等非程序员，把脑内流程变成小软件。

灵感来源：Anthropic hackathon、CrossBeam、PostVisit.ai、v0/Bolt/Lovable。

适用场景：

- 专业人士有强流程，但没有工程团队。
- 他们知道用户痛点，却不会写软件。

MVP demo：

- 用问答把专家流程整理成表单、规则、输出报告。
- 一键发布内部工具。

关键价值：

- AI 编程降低的是表达门槛，不是专业判断门槛。

风险：

- 生成软件质量。
- 数据安全。
- 需要模板化，不然太泛。

### 方向 7：临时互助网络

**一句话**：为一个短时事件自动组群、分工、通知和汇报。

灵感来源：GroupMe、Meshworks、走失搜寻、灾害通信。

适用场景：

- 宠物走失。
- 小区停电。
- 拼车/接送。
- 活动现场协作。
- 校园安全。

MVP demo：

- 输入事件和范围。
- Agent 生成临时群、地图任务、角色分工、进展摘要。

关键价值：

- 微信群能聊天，但不能结构化协作。

风险：

- 滥用、骚扰、隐私。

### 方向 8：机器对机器协商层

**一句话**：当双方都是系统时，不必让人类反复填表和解释。

灵感来源：GibberLink、MCP、Agentic AI Foundation、Conclave。

适用场景：

- 预约。
- 询价。
- 库存确认。
- 保险/理赔进度。
- 商家政策确认。

MVP demo：

- 用户说：“帮我找今晚可带狗的餐厅。”
- 用户 Agent 和商家 Agent 交换宠物体重、时间、座位要求、押金/规则，最后返回可用选项。

关键价值：

- 未来不是每件事都让人填表，agent 可以代表用户交换结构化约束。

风险：

- 标准未统一。
- 身份、授权、审计很重要。

### 方向 9：多 Agent 审议助手

**一句话**：重要决策不是一个 AI 直接拍板，而是多个角色互相挑战后给结论。

灵感来源：Conclave、MetaGPT、ChatDev。

适用场景：

- 投资。
- 医疗就诊选择。
- 旅行/搬家。
- 宠物寄养选择。
- 合同风险。

MVP demo：

- 同一问题由“规划者、风险官、证据官、预算官、用户偏好官”分别给意见。
- 最终输出共识、分歧、证据、建议。

关键价值：

- 用户不只要答案，还要知道答案有没有被反驳过。

风险：

- 多 agent 容易变成表演。
- 必须有真实证据、规则和仲裁。

### 方向 10：个人记忆到行动

**一句话**：不是记录一切，而是把记录变成提醒、建议和自动任务。

灵感来源：Limitless/Rewind、Nyxie、Fitbit。

适用场景：

- 会议待办。
- 家庭事务。
- 宠物健康。
- 慢病管理。
- 亲密关系纪念日/偏好。

MVP demo：

- 用户自然记录：“狗今天又吐了”“医生说两周后复查”。
- Agent 自动关联饮食、用药、活动，生成观察提醒和复诊问题。

关键价值：

- 记忆如果不行动，就只是仓库。

风险：

- 隐私与长期数据安全。

### 方向 11：现场影像到专业建议

**一句话**：把现场照片/视频转成结构化判断与投资/维修/安全建议。

灵感来源：SkySplat、Road footage -> investment recommendations、Good Samaritan。

适用场景：

- 道路养护。
- 小区设施。
- 门店巡检。
- 宠物友好设施评估。
- 无障碍环境评估。

MVP demo：

- 上传一段小区视频。
- Agent 标注风险点：玻璃碎片、施工围挡、无照明、可疑投食点。

关键价值：

- 现实世界大量信息停留在影像里，未结构化。

风险：

- 识别准确率。
- 隐私和公共拍摄合规。

### 方向 12：隐私资格验证

**一句话**：用户不交出原始数据，只证明自己符合条件。

灵感来源：ArcAid、Octagon AI、AI Depression Check。

适用场景：

- 助学资格。
- 保险折扣。
- 宠物领养资格。
- 社区福利。
- 医疗/心理健康筛查。

MVP demo：

- 用户本地上传材料。
- 系统只输出“符合/不符合/缺少材料”，不暴露全部隐私。

关键价值：

- 很多流程需要判断资格，但不应过度收集数据。

风险：

- 技术复杂。
- 生态接受度。

### 方向 13：AI 角色空间化

**一句话**：把 agent 放进可见空间，用户看得见它们在做什么。

灵感来源：AI Town、CharactAR。

适用场景：

- 教育。
- 社区模拟。
- 宠物行为训练。
- 城市规划参与。

MVP demo：

- 一个社区地图里，狗、主人、店铺、物业、志愿者都是可交互角色。
- 用户可以模拟政策或设施变化。

关键价值：

- 多 agent 行为如果只在日志里，很抽象；空间化更好理解。

风险：

- 容易游戏化过度。

### 方向 14：AI 表单终结者

**一句话**：用户提供目标和材料，Agent 负责填表、检查、提交前审阅。

灵感来源：InsuClaimAI、CrossBeam、AutoGPT。

适用场景：

- 理赔。
- 许可证。
- 宠物登记。
- 学校报名。
- 退税。

MVP demo：

- 上传材料，Agent 自动填表并标注不确定项。

关键价值：

- 表单是人类痛苦的大矿。

风险：

- 提交错误责任。
- 需要人工确认。

### 方向 15：知识工作持续雷达

**一句话**：持续追踪某个主题，把新信息按用户关心的问题重排。

灵感来源：ResearchRadar.ai、GPT Researcher、Perplexity。

适用场景：

- 研究者追论文。
- 投资人追公司。
- 宠物主人追某种病/药物/召回。
- 政策从业者追法规变化。

MVP demo：

- 用户定义一个主题和判断框架。
- Agent 每周输出“新增重要变化 + 为什么重要 + 是否影响你”。

关键价值：

- 用户不想看信息流，想知道“这和我有什么关系”。

风险：

- 信息源质量。

### 方向 16：无障碍/适老空间 Agent

**一句话**：给老人、轮椅用户、推婴儿车家庭判断某个地方是否真的可达。

灵感来源：Baymax、Good Samaritan、隐藏信息可视化。

适用场景：

- 商场/医院/公园。
- 公共交通。
- 小区。

MVP demo：

- 用户输入身体约束和目标。
- Agent 给出实际入口、坡道、电梯、休息点、风险。

关键价值：

- 大众地图的“无障碍”标签往往不够具体。

风险：

- 数据采集和更新。

### 方向 17：社区公共问题修复 Agent

**一句话**：把居民零散抱怨转成可提交、可追踪、可协商的公共问题。

灵感来源：FixMyStreet、NASA Space Apps、项飙“附近”。

适用场景：

- 噪音。
- 垃圾。
- 不文明养宠。
- 电梯故障。
- 公园设施损坏。

MVP demo：

- 用户拍照/语音描述问题。
- Agent 定位、归类、收集类似报告、生成给物业/街道的材料。

关键价值：

- 附近不是消费空间，也是协商空间。

风险：

- 政务/物业对接难。
- 社区冲突。

### 方向 18：宠物/儿童/老人友好认证网络

**一句话**：把友好标签从商家自称变成可验证、可更新的信任层。

灵感来源：Airbnb/Rover 信任机制、众包地图、Bunkermode 审计留痕。

适用场景：

- 宠物友好店。
- 亲子友好店。
- 老人友好服务。

MVP demo：

- 用户反馈 + Agent 验证 + 商家确认。
- 展示最近验证时间和具体规则。

关键价值：

- “友好”是细节，不是一个 boolean。

风险：

- 审核成本。
- 商家争议。

### 方向 19：专业副驾驶热线

**一句话**：给人类客服/热线/志愿者实时提示，而不是让 AI 直接面对高风险用户。

灵感来源：TheraSpeak、Good Samaritan。

适用场景：

- 宠物走失热线。
- 心理热线。
- 医疗导诊。
- 法律援助。

MVP demo：

- 通话实时转写。
- Agent 提示下一句问什么、风险等级、转接建议。

关键价值：

- 高风险领域人类仍在回路内。

风险：

- 错误提示。
- 录音隐私。

### 方向 20：空间记忆相册

**一句话**：把用户和附近发生过的事，变成可搜索、可行动的空间记忆。

灵感来源：AI Town、Limitless/Rewind、项飙“附近”。

适用场景：

- 宠物成长。
- 社区关系。
- 家庭活动。
- 城市漫游。

MVP demo：

- 每次出门自动形成“地点 + 事件 + 感受 + 风险 + 人”的记忆。
- 下次推荐时引用：“上次这里 19:30 有大狗，它害怕。”

关键价值：

- 附近不是静态地图，而是个人历史。

风险：

- 隐私。
- 数据采集成本。

### 方向 21：AI 商家前台

**一句话**：小商家没有客服系统，让 Agent 处理常见询问、规则确认、预约和库存。

灵感来源：GibberLink、Nyxie、机器对机器协商。

适用场景：

- 宠物店。
- 餐饮。
- 诊所。
- 培训机构。

MVP demo：

- 用户 Agent 问：“今晚能带 8kg 狗坐外面吗？”
- 商家 Agent 根据规则和库存自动回答，并可人工接管。

关键价值：

- 附近信息过期，很大原因是小商家没有维护渠道。

风险：

- 商家采用慢。
- 错答导致投诉。

### 方向 22：新到一个地方的 72 小时 Agent

**一句话**：搬家、旅行、短住时，快速建立附近生活能力。

灵感来源：Manus/深度研究、附近可用性、life admin。

适用场景：

- 搬家。
- 旅行。
- 留学。
- 带宠入住新城市。

MVP demo：

- 输入新地址和个人/宠物约束。
- Agent 输出 72 小时生存包：医院、超市、宠物医院、可遛区域、禁区、紧急联系人、社区规则。

关键价值：

- 人到新环境时，“附近”最明显地消失。

风险：

- 需要本地信息密度。

### 方向 23：反向 Product Hunt / Idea Radar

**一句话**：不只是发现新产品，而是自动分析它背后的可迁移模式。

灵感来源：Product Hunt、X、GPT Researcher、本报告本身。

适用场景：

- 创业者。
- 投资人。
- 产品团队。
- hackathon 参赛者。

MVP demo：

- 每天抓 X/GitHub/Devpost/Product Hunt。
- 输出“今天 10 个项目 + 模式抽象 + 可迁移行业”。

关键价值：

- 很多人看热闹，但不会提炼模式。

风险：

- 数据抓取与噪音。

### 方向 24：宠物附近 OS

**一句话**：以宠物为入口，重建附近生活的知识、信任和行动网络。

灵感来源：项飙“附近”、Bunkermode、Waze、Rover、Nyxie、GroupMe。

适用场景：

- 宠物出行。
- 宠物应急。
- 宠物服务。
- 宠物社交。
- 社区公共空间。

MVP demo：

- 一个区域内，用户通过模糊目标发起请求。
- Agent 结合地图、宠物档案、社区报告、店铺验证，给出“这次最可用”的方案。
- 同时提供 SOS、反馈、商家确认和附近互助。

关键价值：

- 宠物是很好的入口，因为它天然连接情绪、附近、关系、风险、信任、日常重复。

风险：

- 不能一开始做太大。
- 需要从一个强场景切入。

## 8. 回到宠物：不建议做“路线规划”，建议做 5 个更强切入点

### 切入点 1：宠物 SOS Agent

这是我认为最强的切入之一。

#### 用户场景

- 狗走失。
- 猫从门缝跑出。
- 宠物被咬。
- 疑似中毒。
- 主人临时住院/出差，宠物无人照顾。

#### 为什么痛

- 事件低频，但情绪强、时间敏感、愿意付费。
- 用户慌的时候最需要剧本。
- 附近资源和人非常关键。

#### MVP

输入：

```text
我的狗刚刚在幸福小区东门走失，8kg 柯基，胆小，蓝色牵引绳。
```

输出：

- 自动生成寻宠海报。
- 自动生成小区群/物业/门卫/附近店铺话术。
- 地图标注最后位置、可能路线、搜寻分区。
- 创建目击表单。
- 通知附近愿意接收 SOS 的用户。
- 整理线索时间线。
- 推荐附近 24h 兽医/收容所/派出所联系清单。

#### Agent 价值

- 不是“给建议”，而是“替你启动行动系统”。

#### 黑客松演示方式

- 模拟一只狗走失。
- 30 秒内生成海报、地图、通知、任务分区、线索后台。
- 让评委看到“恐慌 -> 有序行动”的强对比。

### 切入点 2：附近可用性 Agent

这比路线规划更贴近你说的“模糊目标”。

#### 用户场景

```text
今晚想带狗出门透透气，别太远，能坐一下，最好顺路买点药。
```

#### 为什么痛

- 大众地图只会告诉你“哪里有店”。
- 用户真正关心的是“这只狗、这个时间、这次目的，哪里最不容易踩坑”。

#### MVP

一个小区域内收集：

- 宠物友好店铺。
- 具体规则。
- 最近验证时间。
- 水碗/户外位/空间大小。
- 拥挤度。
- 大狗密度。
- 附近药店/医院。
- 用户反馈。

Agent 输出：

- 3 个“可用方案”，不是 3 条路线。
- 每个方案有适合理由、风险、证据、备选。
- 一键询问店铺确认。

#### 关键差异

大众地图：

```text
附近宠物友好咖啡店
```

你的 Agent：

```text
今晚 8 点，带一只胆小的 8kg 狗，想坐 30 分钟并买驱虫药，哪一套附近行动最稳？
```

### 切入点 3：宠物友好真实性层

#### 用户场景

- 点评写“宠物友好”，去了发现只能户外。
- 商场规则变了。
- 店员换班，态度不同。
- 大众点评信息过期。

#### 产品

不是地图，而是一套真实性机制：

- 最近一次用户验证。
- 最近一次商家确认。
- 规则细节。
- 例外情况。
- 被拒记录。
- 照片证据。
- Agent 主动复核过期信息。

#### 为什么有价值

“宠物友好”不是 yes/no，而是：

- 多大的狗？
- 是否必须牵绳？
- 室内还是室外？
- 是否要包？
- 是否能上椅子？
- 是否有其他狗？
- 雨天怎么办？

### 切入点 4：新养宠/新搬家 72 小时附近包

#### 用户场景

- 第一次养狗。
- 刚搬家。
- 带宠旅行。
- 换城市。

#### 产品

输入地址和宠物信息，Agent 输出：

- 最近靠谱宠物医院。
- 夜间急诊。
- 可遛区域。
- 禁入区域。
- 宠物店/洗护/寄养。
- 小区规则。
- 附近狗主人社群。
- 疫苗/登记/保险待办。
- 常见风险点。

#### 为什么成立

这是“附近消失”最强的时刻：一个人带着宠物进入陌生环境，最缺本地生活知识。

### 切入点 5：邻里信任型宠物互助

#### 用户场景

- 临时加班，需要人遛狗。
- 出差 2 天，需要喂猫。
- 狗走失，需要附近人帮看。
- 大狗/小狗社交需要靠谱搭子。

#### 产品

Agent 做：

- 筛选附近可信人。
- 明确任务边界。
- 生成约定。
- 照片/轨迹回传。
- 支付/押金。
- 异常上报。
- 评价和关系沉淀。

#### 为什么比普通社区群强

微信群有关系，但没有结构；平台有结构，但缺附近关系。Agent 可以在二者之间做一个轻量信任层。

## 9. 推荐的 MVP：不要从地图开始，从“附近可用性 + SOS”开始

### 9.1 MVP 命题

建议第一版产品叫法可以是：

> 宠物附近 Agent：帮你判断这次出门/应急/求助，附近什么是真的可用。

### 9.2 第一阶段只做一个城市里的一个小区域

不要一开始全城。选一个宠物密度高、店铺集中、用户愿意反馈的区域，比如：

- 上海的一个宠物友好街区。
- 北京某个大型社区周边。
- 深圳/杭州一个年轻养宠人聚集区。

### 9.3 功能范围

第一阶段只做 4 件事：

1. **模糊目标问答**
   - 用户说目标和宠物约束。
   - Agent 给出 2-3 个附近行动方案。

2. **真实性验证**
   - 每个地点显示最近验证时间、规则细节、证据。
   - 信息过期时，Agent 提示“需要确认”。

3. **一键确认**
   - Agent 生成给店铺/医院/服务者的询问话术。
   - 早期可以让用户手动复制，后期再自动发消息/电话。

4. **SOS 剧本**
   - 走失海报。
   - 目击表单。
   - 附近通知。
   - 搜寻时间线。

### 9.4 典型用户流

```text
用户：今晚 8 点想带奶茶出去 40 分钟。它胆小，怕大狗，最好能坐一下，顺路买驱虫药。

Agent：
1. 方案 A：森屿咖啡户外位 + 乐康宠物药房
   - 适合理由：距离 700m；户外位最近 2 天被验证可带中小型犬；20:00 后人少；药房有驱虫药库存。
   - 风险：咖啡店下雨会关闭户外位；建议先确认。
   - 动作：一键询问店铺 / 导航 / 加入反馈。

2. 方案 B：河边步道 + 自助宠物柜
   - 适合理由：狗密度低；无店铺停留压力。
   - 风险：19:30-20:30 有夜跑人群，胆小狗可能紧张。

3. 不推荐：星悦广场
   - 原因：最近 3 次反馈显示保安对宠物入内执行不稳定；周六大狗聚集。
```

这比路线规划强，因为它回答的是：**这次生活目标能不能在附近顺利完成。**

### 9.5 黑客松 demo 脚本

一个 3 分钟 demo 可以这样设计：

1. 展示普通地图搜索“宠物友好咖啡店”，结果很多但无法判断。
2. 用户输入模糊目标：“胆小狗、今晚、坐一会儿、顺路买药。”
3. Agent 展示宠物档案：胆小、怕大狗、喜欢安静、上次在某店被拒。
4. Agent 读取地点规则、用户反馈、天气、营业时间、药店库存。
5. 输出 3 个方案，并标注证据与置信度。
6. 用户点击“一键确认”，生成给店铺的话术。
7. 突发切换：用户点击“狗走失 SOS”。
8. Agent 生成寻宠海报、通知范围、线索表单、搜寻分区。

这个 demo 同时覆盖：

- 模糊目标。
- 附近可用性。
- 真实性。
- Agent 行动。
- SOS 高情绪场景。

## 10. 数据与护城河

宠物附近产品的护城河不应是“地点列表”，而是下面这些更难复制的数据：

### 10.1 具体规则数据

不是：

```text
宠物友好：是
```

而是：

```text
允许 10kg 以下犬进入户外位；必须牵绳；雨天户外位关闭；周末 14:00-17:00 店内拥挤；最近确认：2026-06-12。
```

### 10.2 宠物约束画像

包括：

- 体型。
- 胆量。
- 对狗/人/车/噪音反应。
- 健康限制。
- 主人偏好。
- 历史踩坑。

### 10.3 本地实时反馈

包括：

- 今天是否被拒。
- 是否有大狗聚集。
- 是否施工。
- 是否有水碗。
- 是否有冲突。
- 是否拥挤。

### 10.4 行动结果

这很关键：

- Agent 推荐后，用户是否真的去了？
- 是否顺利？
- 是否被拒？
- 是否修改了方案？
- 哪些证据最有用？

这会让系统从“信息地图”变成“决策系统”。

### 10.5 信任关系

包括：

- 谁帮助过谁。
- 谁可靠。
- 谁适合照看什么宠物。
- 哪些店铺响应快。
- 哪些医院夜间真的可用。

## 11. 商业模式假设

### 11.1 C 端订阅

适合：

- 高价值功能：SOS、健康记录、问诊准备、出行确认。

问题：

- 单纯地图很难订阅。
- 必须有强情绪或高频价值。

### 11.2 交易抽佣

适合：

- 寄养。
- 遛狗。
- 上门喂养。
- 洗护。
- 宠物接送。
- 保险/问诊。

问题：

- 进入服务交易后，平台责任上升。

### 11.3 商家 SaaS / 前台 Agent

适合：

- 宠物店。
- 宠物医院。
- 宠物友好餐饮。
- 寄养机构。

提供：

- 自动问答。
- 规则维护。
- 预约。
- 库存。
- 用户画像。
- 活动触达。

问题：

- 小商家付费意愿不一。
- 需要先有用户流量。

### 11.4 社区/物业合作

适合：

- 宠物密度高的小区。
- 商圈。
- 公园管理。

提供：

- 文明养宠工具。
- 走失应急系统。
- 投诉与反馈整理。
- 宠物友好设施建议。

问题：

- 销售周期长。

### 11.5 数据/API

适合后期：

- 地图平台。
- OTA。
- 酒店。
- 商场。
- 本地生活平台。

提供：

- 宠物友好规则 API。
- 附近风险 API。
- 宠物出行适配 API。

问题：

- 前期数据密度不足。

## 12. 风险与反模式

### 12.1 X 传播不等于产品成立

GibberLink、AI Town、draw-a-ui 等很容易传播，因为 demo 魔法感强。但产品化需要回答：

- 用户第二次为什么回来？
- 是否有真实付费意愿？
- 错误成本是什么？
- 数据如何持续更新？

### 12.2 黑客松获奖不等于能商业化

黑客松强调 36-48 小时内可 demo。真实产品还需要：

- 稳定性。
- 数据质量。
- 运营。
- 合规。
- 客服。
- 纠纷处理。

但 hackathon 项目也不是完全一次性。关于 Devpost 项目的研究显示，hackathon 代码有一部分会被后续复用，说明它可以成为长期产品的起点。

### 12.3 Agent 安全风险

如果 Agent 能浏览网页、读评论、发消息、付款、预约，它就面对 prompt injection 和高权限误操作风险。Web-use agents 的安全研究已经指出：网页评论、广告、文档等不可信内容可能诱导 agent 偏离用户目标。

对宠物附近 Agent 来说，必须设计：

- 高风险动作前人工确认。
- 外部网页/点评内容隔离。
- 来源置信度。
- 操作日志。
- 支付/钥匙/定位等敏感动作的权限分级。
- 医疗建议边界。

### 12.4 地图冷启动

宠物友好地图最怕“空”。解决方式不是一次性铺全国，而是：

- 从一个高密度区域开始。
- 先做强场景，比如 SOS 和附近可用性。
- 用 Agent 主动验证过期信息。
- 用户每次完成行动后顺手反馈。

### 12.5 过度规划

不要让产品变成“完美路线 planner”。用户生活不是物流优化。更好的体验是：

- 给 2-3 个稳妥方案。
- 标出不确定性。
- 提供备选。
- 支持现场变化。

## 13. 优先级建议

### 13.1 最值得做的方向

我建议优先级如下：

| 优先级 | 方向 | 理由 |
|---|---|---|
| P0 | 宠物 SOS Agent | 情绪强、行动明确、黑客松 demo 强、愿意付费 |
| P0 | 附近可用性 Agent | 与“附近的消失”结合最好，能区别于普通地图 |
| P1 | 宠物友好真实性层 | 是附近可用性的底层数据资产 |
| P1 | 新搬家/新养宠 72 小时包 | 新环境下痛点集中，容易转化 |
| P2 | 邻里信任互助 | 护城河强，但运营和责任更重 |
| P2 | 健康/行为日志 | 长期价值强，但录入负担要小心 |
| P3 | 纯路线规划 | 差异弱，容易被大地图覆盖 |

### 13.2 最小可行切口

不要第一版做完整 App。可以先做：

```text
一个微信/网页轻工具：
1. 输入宠物档案 + 模糊目标
2. 返回附近 3 个可用方案
3. 支持一键生成店铺确认话术
4. 支持走失 SOS 海报与线索表单
5. 用户反馈后更新地点规则
```

### 13.3 7 天原型计划

第 1 天：选定一个区域，人工收集 30 个地点。  
第 2 天：定义地点 schema：规则、验证时间、风险、证据、反馈。  
第 3 天：做宠物档案与模糊目标解析。  
第 4 天：做推荐解释与置信度。  
第 5 天：做一键确认话术和反馈入口。  
第 6 天：做走失 SOS 海报、线索表单、通知清单。  
第 7 天：找 5 个养宠用户试用，记录他们真正问的问题。

### 13.4 4 周验证计划

第 1 周：人工 concierge 验证 20 次“附近可用性”请求。  
第 2 周：上线最小网页工具，支持地点反馈和规则更新。  
第 3 周：上线 SOS 剧本，跑模拟演练。  
第 4 周：与 5-10 家宠物友好店/宠物医院建立验证机制。

### 13.5 成功指标

比“打开地图次数”更好的指标：

- 模糊目标请求完成率。
- 用户是否采纳推荐。
- 推荐后是否顺利完成目标。
- 信息被证伪率。
- 地点规则更新频率。
- SOS 剧本启动到关键动作完成时间。
- 用户二次请求率。
- 商家确认响应率。

## 14. 一句话结论

如果继续做宠物方向，我不建议把核心定义成“遛狗路线规划”。更好的定义是：

> **宠物附近 Agent：把附近重新变成可感、可信、可行动的生活网络。**

路线只是其中一个输出；真正的产品内核是：

- 理解模糊目标。
- 记住宠物约束。
- 验证附近信息。
- 判断这一次是否可用。
- 替用户协调。
- 在紧急时启动行动剧本。
- 把每次行动沉淀成附近知识。

这比“宠物友好地图”厚，也比“路线推荐”更接近痛点。它同时能吸收 X/黑客松里最强的创意模式：魔法输入、行动闭环、危机响应、多 agent 审议、机器协商、众包真实世界数据，以及对“附近”的重新组织。
