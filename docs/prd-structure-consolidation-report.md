# Pet Mobility Agent PRD 结构合并建议报告

日期：2026-06-13

## 一句话结论

当前 PRD 的问题不是内容过多，而是标题层级过密、章节粒度不一致、产品叙事和工程落地信息混在一起。建议把 11 个一级章节压缩为 6 个一级章节，把大量三级标题改成表格、列表或模块内字段说明，让读者按“为什么做 -> 给谁用 -> MVP 做什么 -> 怎么用 -> 怎么实现 -> 如何可信”一路读下来。

推荐目标：

- 一级章节：从 11 个降到 6 个。
- 目录节点：从当前 66 个标题节点降到 35 个以内。
- 三级标题：大部分改成表格或模块内短段落，只保留真正需要被目录索引的关键流程。

## 多子 agent 结论汇总

本次让 3 个子 agent 分别从产品叙事、文档信息架构、MVP 工程落地三个角度评估，结论高度一致：

| 视角 | 核心判断 | 具体建议 |
| --- | --- | --- |
| 产品叙事 | PRD 像功能资料库，不像连续产品判断。 | 把“产品介绍 + 背景”合并成“产品定位与机会判断”；把尾部的隐私、非功能、风险合并。 |
| 信息架构 | 中段标题密度最高，模块说明和用户流程重复。 | 降低三级标题，把相似内容改成表格；把命名口径和版本规则移出正文主线。 |
| 工程落地 | 宠物分身和社区热力容易变成并列主线，稀释 MVP。 | 以“常走路线 + 宠物档案 + 风险门控 + POI 可信度 + 轻量反馈”为 MVP 主线。 |

我的综合判断：这份 PRD 应优先采用“产品叙事版结构”，因为它现在仍处在方向定义阶段；但在每个核心模块里保留工程可执行字段，避免变成纯概念文档。

## 当前结构问题

### 1. 开头被文档管理信息打断

当前开头是：

```text
产品介绍
  产品定位
  产品命名口径
  发布与版本规则
背景
```

`产品命名口径` 和 `发布与版本规则` 不是用户、市场、产品或 MVP 判断，会打断读者进入产品主线。它们更适合放到：

- `.qmd` front matter。
- PRD 末尾的“文档维护说明”。
- 或从正文删除，仅在仓库 README / AGENTS 指南中维护。

### 2. “为什么做”被拆成太多小标题

`产品定位`、`市场与用户背景`、`核心问题`、`类比产品启发`、`机会判断`本质上都在回答同一个问题：

> 为什么现在需要一个懂这只宠物的附近地图？

建议合并为一个一级章节，内部用 3-4 个二级标题承接即可。

### 3. “核心产品模块”承载过重

当前 `核心产品模块` 内部有宠物档案、宠物分身、地图图谱、常走路线、出门 Agent、社区热力、UGC POI，几乎等于整份产品设计。问题在于：

- 宠物分身放得太靠前，会让读者误以为它和出门判断同等核心。
- 地图图谱、社区热力、UGC POI 都是“地图信号供给”，可以合并讲。
- 出门 Agent 推荐和常走路线才是核心差异化，应在模块顺序里更靠前。

### 4. 模块章节和流程章节重复

`核心产品模块` 已经讲了很多输入、输出、流程和规则，后面的 `核心用户流程` 又以 Mermaid 重新讲一遍，导致阅读上出现“刚看过一遍，又换形式看一遍”的感觉。

建议只保留一个“核心体验闭环”章节，流程围绕 3 个主路径：

- 首次使用。
- 无目标日常出门 / 有目标附近出行。
- 地图贡献与出行后反馈。

宠物分身生成流程可以放到“宠物分身与状态表达”模块内，不必作为主流程进入目录。

### 5. 系统支撑信息拆得过散

`Agent 架构`、`数据模型`、`事件采集`都在回答同一个问题：

> 产品判断如何被系统支撑？

建议合并成 `Agent 架构与数据`。先说 MVP 不做真正多 Agent，而是单 Orchestrator + 确定性能力模块；再说核心对象和事件。

### 6. 尾部约束章节可以收束

`隐私与安全`、`非功能需求`、`风险与应对`都属于可信边界。拆成三个一级标题会显得尾部很碎。建议合并为 `可信边界与风险`，内部用表格收纳：

- 位置与路线隐私。
- 社区热力脱敏。
- 推荐安全边界。
- 性能、可用性、可解释性。
- 风险与应对。

## 推荐目录结构

我建议采用 6 个一级章节：

```markdown
# 产品定位与机会
## 一句话定位
## 用户背景与核心问题
## 类比启发与机会判断

# 用户场景与 MVP 范围
## 用户分层
## 核心使用场景
## P0：必须跑通的产品闭环
## 后续路线图与暂不进入范围

# 核心体验闭环
## 首次使用
## 出门推荐：无目标与有目标
## 地图贡献与反馈沉淀

# 核心能力设计
## 宠物档案
## 常走路线记忆
## 出门 Agent 推荐
## 地图 POI、风险点与社区信号
## 宠物分身与状态表达

# Agent 架构与数据
## MVP 架构
## 核心对象
## 事件采集

# 可信边界与风险
## 位置与路线隐私
## 社区热力脱敏
## 推荐安全与可解释性
## 非功能需求
## 风险与应对
```

这个结构的优点：

- 开头更快进入产品主张。
- P0 范围被前置，避免后文功能发散。
- 用户流程和能力模块分离：先讲用户怎么用，再讲系统需要什么能力。
- 宠物分身被放回“状态表达与反馈入口”，不会抢走“常走路线 + 出门判断”的主线。
- 隐私、安全、非功能和风险统一收束到可信边界。

## 不建议合并或弱化的内容

以下内容应该继续保留清晰位置：

| 内容 | 原因 |
| --- | --- |
| P0 闭环 | 这是防止产品发散的范围合同。 |
| 常走路线记忆 | 这是区别于普通地图和 POI 列表的关键差异点。 |
| 出门 Agent 推荐 | 这是产品从“找地点”升级为“附近判断”的核心。 |
| hard veto / 推荐安全边界 | 宠物出行存在安全风险，不能被推荐分抵消。 |

| UGC POI 可信度 | 没有可信度机制，宠物友好地图会退化成普通点评列表。 |

## 需要特别调整的产品主线

### 1. 宠物分身要降权，但不能删除

当前宠物分身章节内容很丰富，容易让 PRD 看起来像“虚拟宠物 + 地图”的双主线。建议重新定义为：

> 宠物分身是宠物画像、推荐解释和反馈引导的可视化身体。

P0 只写：

- Q 版 2D 头像。
- 3 类状态表达：天气风险、路线强度、遇狗概率或 POI 状态。
- 生成失败降级。
- 不阻塞出门推荐。

3D、rig、动画、GLB/USDZ、供应商路线可以放入附录或 P1/P2，不要在主 PRD 中占太多篇幅。

### 2. 社区热力要从“功能图层”改为“信号输入”

当前社区热力图容易被读成一个独立社交功能。建议拆清：

- P0：社区趋势可以作为小范围 pilot、模拟数据或低粒度聚合信号进入推荐。
- P1：才做面向用户展示的热力图层。

这样能避免 MVP 被热力图可视化拖住。

### 3. Dog Need Fulfillment Graph 要换成更工程可读的表达

`Dog Need Fulfillment Graph` 是一个有启发的概念，但在 PRD 主目录里略抽象。建议在正文中改为：

> 宠物友好地图数据结构

然后在段落中说明它本质上是一个图谱。工程字段可直接写成：

- `Route`
- `POI`
- `RiskPoint`
- `HeatCell`
- `Observation`
- `Rule`
- `RecommendationContext`

## 可选工程评审版目录

如果下一步要直接给工程团队评审，可以采用更开发向的 8 章结构：

```markdown
# MVP 目标与非范围
# 核心用户流程
# 能力模块
# 推荐规则与验收口径
# 数据模型与事件
# 隐私与安全
# 非功能需求
# 风险与后续路线图
```

工程版更短、更硬，但会牺牲一些产品叙事。当前阶段我不建议直接切到工程版，除非目标是马上拆任务开发。

## 推荐执行顺序

1. 先把正文主目录重排为 6 个一级章节。
2. 把 `产品命名口径` 和 `发布与版本规则` 移出主线。
3. 把 `核心用户流程` 压缩为 `核心体验闭环`，删除和模块重复的流程细节。
4. 把 `社区热力图 + UGC POI + 宠物友好地图图谱` 合并为地图信号模块。
5. 把 `Agent 架构 + 数据模型 + 事件采集` 合并。
6. 把 `隐私与安全 + 非功能需求 + 风险与应对` 合并。
7. 最后统一检查目录层级，把规格性三级标题改为表格。

## 最终建议

建议不要简单地“删标题”，而是把 PRD 改成一条更强的判断链：

```text
这是什么
-> 为什么现在有机会
-> 谁在什么场景下需要
-> MVP 必须证明什么
-> 用户怎么完成一次出门判断
-> 产品需要哪些核心能力
-> Agent 和数据如何支撑
-> 隐私、安全、风险如何兜底
```

按这个方向改完后，PRD 会更像一个可以被投资人、产品、设计和工程共同阅读的文档，而不是功能规格的堆叠。
